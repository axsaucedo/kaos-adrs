# KAOS-E5-3 — Langfuse

Deep dive on Langfuse as the candidate platform for the results-and-online-eval half of KAOS evals: trace storage for the Pydantic AI runtime (OTLP), datasets/experiments as the shared results store, and LLM-as-judge over live traces. All claims verified against the official docs ([langfuse.com/docs](https://langfuse.com/docs), [langfuse.com/self-hosting](https://langfuse.com/self-hosting)) and the [GitHub repo](https://github.com/langfuse/langfuse) as of 2026-07.

## 1. OpenTelemetry ingestion

- Langfuse exposes a native OTLP endpoint at `/api/public/otel` (plus the signal-specific `/api/public/otel/v1/traces` for collectors that require it). It supports OTLP over HTTP with both `http/json` and `http/protobuf`; **gRPC is not supported**. Auth is HTTP Basic with base64-encoded `pk-lf-...:sk-lf-...` project keys ([OTel get started](https://langfuse.com/docs/opentelemetry/get-started)).
- Incoming spans are mapped onto Langfuse's trace/observation model via a documented [property-mapping table](https://langfuse.com/docs/opentelemetry/get-started): trace-level attributes (`langfuse.user.id`/`user.id`, `langfuse.session.id`/`session.id`, tags, version, release, trace input/output) and observation-level attributes (type, level, input/output, and for generations: model name, model parameters, token usage, cost). OTel GenAI semantic-convention attributes are recognized for the generation fields.
- Unmapped span attributes are preserved under `metadata.attributes` and resource attributes under `metadata.resourceAttributes`; the `langfuse.trace.metadata.*` / `langfuse.observation.metadata.*` prefixes produce filterable metadata ([mapping docs](https://langfuse.com/docs/opentelemetry/get-started)).
- Pydantic AI has a [documented first-class path](https://langfuse.com/integrations/frameworks/pydantic-ai): set `LANGFUSE_PUBLIC_KEY`/`LANGFUSE_SECRET_KEY`/`LANGFUSE_BASE_URL`, initialize the Langfuse client, and enable Pydantic AI's built-in OTel instrumentation (`Agent.instrument_all()` or `instrument=True` per agent). This uses Pydantic AI's native OTel span export — no Logfire backend or Logfire account is required, which matches the KAOS runtime's existing OTLP export. Equivalently, KAOS could keep a plain OTLP exporter (or an OTel Collector) pointed at `/api/public/otel` with the Basic-Auth header.

**Fit for KAOS:** the runtime already emits OTel GenAI spans over OTLP/HTTP, so ingestion is a per-namespace exporter config (endpoint + project keys), not an SDK adoption.

## 2. Datasets and experiments

- [Datasets](https://langfuse.com/docs/evaluation/dataset-runs/datasets) are project-scoped collections of items (`input`, `expected_output`, `metadata`), manageable via UI, SDK (`langfuse.create_dataset`, `create_dataset_item`), and the public REST API. Items support JSON-Schema validation, folder-style organization, and automatic versioning of every mutation for reproducible runs.
- Dataset items can be **created directly from production traces** by linking a trace or observation ID, which is the documented loop for turning failing production cases into regression tests ([datasets docs](https://langfuse.com/docs/evaluation/dataset-runs/datasets)).
- Experiment runs ("dataset runs") link one trace per dataset item to a named run; runs are compared side-by-side in the UI with item-level scores and run-level aggregate metrics ([remote dataset runs](https://langfuse.com/docs/evaluation/dataset-runs/remote-run)).
- Crucially for KAOS, [remote dataset runs](https://langfuse.com/docs/evaluation/dataset-runs/remote-run) explicitly support an **external harness**: Langfuse can trigger it via webhook (Custom Experiment `Run` button sends dataset ID/name + custom config), the harness fetches items via SDK/API, executes the app, links traces to items, and pushes scores back — runs then appear in the normal UI comparison view. This is exactly the shape of a KAOS-owned Pydantic-Evals harness reporting into Langfuse.

## 3. Evaluation

- [LLM-as-a-judge evaluators](https://langfuse.com/docs/evaluation/evaluation-methods/llm-as-a-judge) run on three targets: live observations, live traces, and experiment/dataset runs — so the same evaluator catalog covers both online eval and offline experiments. Configuration (judge model, variable mapping from trace/item fields, prompt preview, sampling/filtering) is via UI or programmatically via the public API.
- **Judge model via LiteLLM works.** Judge models are configured through project-level [LLM connections](https://langfuse.com/docs/administration/llm-connection), which natively support OpenAI, Azure OpenAI, Anthropic, Google (AI Studio/Vertex), and Bedrock, and explicitly support any OpenAI-compatible gateway (LiteLLM is named) by selecting the OpenAI adapter and setting a custom **Base URL** plus custom model names. Constraint: the gateway must support tool calling in the OpenAI format, since judges rely on structured output — the KAOS LiteLLM ModelAPI satisfies this for tool-calling-capable models.
- Langfuse ships a catalog of [managed evaluator templates](https://langfuse.com/docs/evaluation/evaluation-methods/llm-as-a-judge) (maintained by Langfuse and partners such as Ragas) covering hallucination, context relevance, toxicity, helpfulness, etc.; custom templates are supported.
- **Custom scores** from external code use the [scores API/SDK](https://langfuse.com/docs/evaluation/evaluation-methods/custom-scores): `POST /api/public/scores` with `traceId` (or `sessionId`), optional `observationId`, `name`, `value`, and `dataType` in `NUMERIC | CATEGORICAL | BOOLEAN | TEXT`, plus optional `comment` and `configId` for schema-enforced scores; a stable `id` gives idempotent upserts. This is the write path for a Pydantic-Evals harness.
- [Human annotation queues](https://langfuse.com/pricing-self-host) provide managed workflows for manual scoring of traces/observations and are part of the free OSS tier.
- **Licensing of evals features:** on 2025-06-04 Langfuse [open-sourced all remaining product features under MIT](https://langfuse.com/changelog/2025-06-04-open-sourcing-langfuse) ("Doubling Down on Open Source", [blog](https://langfuse.com/blog/2025-06-04-open-sourcing-langfuse-product)) — managed LLM-as-a-judge, annotation queues, prompt experiments, and the playground are free to self-host on versions >3.65. Per the [self-host pricing page](https://langfuse.com/pricing-self-host), what remains commercially licensed is not evals-related: project-level RBAC, audit logs, SCIM, data-retention management, UI customization, and the organization/instance management APIs.

## 4. Self-hosting footprint

- [Architecture](https://langfuse.com/self-hosting): two application containers — **langfuse-web** (UI + APIs) and **langfuse-worker** (async event processing) — plus four storage systems: **Postgres** (transactional/config data), **ClickHouse** (OLAP store for traces, observations, scores), **Redis/Valkey** (queues + cache), and **S3-compatible blob storage** (raw events, multi-modal attachments; MinIO if in-cluster). An LLM gateway (the KAOS LiteLLM ModelAPI) plugs in for playground/judge features. Self-hosted runs the same codebase as Langfuse Cloud.
- [Minimum sizing guidance](https://langfuse.com/self-hosting/configuration/scaling): web 2 CPU/4 GiB, worker 2 CPU/4 GiB, Postgres 2 CPU/4 GiB, Redis 1 CPU/1.5 GiB, ClickHouse 2 CPU/8 GiB (16 GiB+ recommended for larger deployments), MinIO ~2 CPU/4 GiB — roughly **12 CPU / 25 GiB as a floor** across six stateful-ish services, three of which (Postgres, ClickHouse, MinIO) need persistent volumes. Scaling docs cover worker autoscaling on queue length and splitting ingestion vs UI web deployments.
- The official [Kubernetes Helm chart](https://langfuse.com/self-hosting/deployment/kubernetes-helm) ([langfuse/langfuse-k8s](https://github.com/langfuse/langfuse-k8s)) deploys the app containers and, by default, bundles Postgres, ClickHouse, and Redis (each can be switched to external instances via `values.yaml`). It is actively maintained by the Langfuse team and is the documented production path; Docker Compose exists but is explicitly non-HA/testing-only ([self-hosting overview](https://langfuse.com/self-hosting)).
- Versioning/upgrades: v3 is the current major line with frequent minor releases (616 releases; [v3.214.0 on 2026-07-15](https://github.com/langfuse/langfuse/releases)); background migrations are used to minimize upgrade downtime ([self-hosting docs](https://langfuse.com/self-hosting)).

**Honest comparison:** this is materially heavier than anything KAOS deploys today — ClickHouse alone is a bigger operational commitment than the entire KAOS control plane, and Langfuse v3 cannot run in a "Postgres-only" mode. For platform teams already running observability infra this is acceptable; for a minimal KAOS install it is not a default-on dependency.

## 5. APIs for a KAOS integration

- **Write path (external harness → Langfuse):** [public REST API](https://api.reference.langfuse.com/) with project-scoped Basic Auth covers datasets and items (`/api/public/v2/datasets`, `/api/public/dataset-items`), dataset run items linking traces to items (`/api/public/dataset-run-items`), and scores (`/api/public/scores`). The [remote-run flow](https://langfuse.com/docs/evaluation/dataset-runs/remote-run) formalizes this for harnesses that execute outside Langfuse.
- **Read path (Langfuse → KAOS):** the same API exposes GET endpoints for datasets, runs (`/api/public/datasets/{name}/runs`), scores, and traces, so a KAOS controller *could* poll run scores and reflect pass/fail in CRD status.
- **Recommendation:** KAOS should own canonical pass/fail. The harness (Pydantic Evals) computes verdicts and writes them to CRD status directly, then mirrors scores/runs into Langfuse for exploration and comparison. Reading verdicts back out of Langfuse would make Kubernetes reconciliation depend on an external stateful platform's availability and on eventual-consistency of async ingestion (events are queued through S3/Redis before landing in ClickHouse per the [architecture docs](https://langfuse.com/self-hosting)), which is the wrong direction of coupling for a controller.

## 6. Multi-tenancy

- Model: **organizations → projects → users with roles** (Owner/Admin/Member/Viewer/None), with org-level roles inherited by projects ([RBAC docs](https://langfuse.com/docs/administration/rbac)). API keys are **project-scoped**, so a namespace-per-team KAOS cluster maps cleanly to project-per-namespace (or project-per-team): each namespace's OTLP exporter and harness get that project's key pair, and data isolation follows from key scoping.
- Caveats for the OSS tier: fine-grained **project-level role overrides are Enterprise-gated** on self-host, as is the **organization management API** for programmatically provisioning orgs/projects ([self-host pricing](https://langfuse.com/pricing-self-host)). On MIT, KAOS would create projects manually (or via the UI/owner session) rather than automating tenant provisioning, and RBAC granularity stops at org-level roles.

## 7. License and maturity

- **MIT core with `ee` folders excluded**: "This repository is MIT licensed, except for the `ee` folders" ([repo README](https://github.com/langfuse/langfuse)). Since June 2025 all product features (evals included) are MIT; only enterprise security/platform features listed above remain commercial ([changelog](https://langfuse.com/changelog/2025-06-04-open-sourcing-langfuse)).
- Traction and cadence: ~31k GitHub stars, 616 releases, near-daily release cadence ([GitHub](https://github.com/langfuse/langfuse)).
- Backing: Langfuse (Y Combinator W23) was **acquired by ClickHouse, Inc. in January 2026** alongside ClickHouse's $400M Series D; both companies publicly committed that Langfuse stays open source and self-hostable with no planned licensing changes and the team continues building the product ([Langfuse announcement](https://langfuse.com/blog/joining-clickhouse), [ClickHouse announcement](https://clickhouse.com/blog/clickhouse-acquires-langfuse-open-source-llm-observability)). This strengthens long-term viability, with the usual acquisition caveat that roadmap incentives now favor ClickHouse-adjacent deployments.

## How it would embed in KAOS

- **Deploy-alongside, optional dependency — not a hard one.** Ship Langfuse as an opt-in platform component (Helm sub-chart or documented companion install), the same pattern as other KAOS platform dependencies. Core KAOS eval flows (offline Pydantic-Evals runs, CRD pass/fail, gating) must work with Langfuse absent; when present, KAOS wires OTLP export per namespace and the harness mirrors datasets, runs, and scores into the matching Langfuse project.
- **What KAOS owns regardless:** the offline harness and its dataset format (source of truth in Git/CRDs, pushed to Langfuse datasets rather than authored there), verdict computation and thresholds, CRD status as the canonical pass/fail record, and the LiteLLM ModelAPI that both agents and Langfuse judge evaluators call through.
- **What Langfuse owns when installed:** trace storage and exploration, online LLM-as-judge over production traces, annotation queues, run-comparison UI, and the trace-to-dataset-item curation loop.

### Risks / gaps

- **Operational weight:** six-service footprint (~12 CPU / 25 GiB minimum, ClickHouse + Postgres + Redis + MinIO with persistent volumes) dwarfs the rest of a KAOS install; must stay optional ([scaling docs](https://langfuse.com/self-hosting/configuration/scaling)).
- **EE drift:** evals are MIT today, but the EE boundary has moved before; multi-tenant conveniences KAOS wants (project-level RBAC, org management API for automated tenant provisioning) are already Enterprise-gated ([pricing](https://langfuse.com/pricing-self-host)). The ClickHouse acquisition adds mild uncertainty despite public no-relicensing commitments.
- **Coupling CRD status to an external platform:** reading pass/fail from Langfuse would put an async, eventually-consistent OLAP pipeline in the reconciliation path; keep verdicts in KAOS and treat Langfuse as a mirror ([architecture](https://langfuse.com/self-hosting)).
- **No gRPC OTLP:** exporters must use OTLP/HTTP; clusters standardized on OTLP/gRPC need an OTel Collector hop ([OTel docs](https://langfuse.com/docs/opentelemetry/get-started)).
- **Manual tenant provisioning on MIT:** no org/project provisioning API in the free tier means namespace-to-project bootstrap is a manual or UI-driven step ([pricing](https://langfuse.com/pricing-self-host)).
