# KAOS-E5-2 â€” DeepEval

DeepEval ([confident-ai/deepeval](https://github.com/confident-ai/deepeval)) is an Apache-2.0 "pytest for LLMs" evaluation framework maintained by Confident AI. It runs evaluations locally and positions the Confident AI SaaS as an optional layer for reports, dataset hosting, and trace observability ([getting started](https://deepeval.com/docs/getting-started)). For KAOS it is the harness alternative to Pydantic Evals and, more realistically, a supplementary metric library â€” its ~40 research-backed metrics are the main draw, while its dataset/reporting/tracing surfaces pull toward the SaaS.

## Core contract

- **Test cases.** [`LLMTestCase`](https://deepeval.com/docs/evaluation-test-cases) is the single-turn unit with nine fields: mandatory `input` and `actual_output`, plus optional `expected_output`, `context` (ground-truth strings), `retrieval_context` (RAG retriever output), `tools_called`, `expected_tools`, `token_cost`, and `completion_time`. Agentic fields use a `ToolCall` object (`name` required; `description`, `reasoning`, `output`, `input_parameters` optional). Multi-turn evaluation uses `ConversationalTestCase` built from turns, and multimodal cases embed `MLLMImage` objects.
- **Goldens and datasets.** An [`EvaluationDataset`](https://deepeval.com/docs/evaluation-datasets) is "a collection of goldens" â€” `Golden` (single-turn) or `ConversationalGolden` (multi-turn) objects holding inputs and expected results but no dynamic outputs. Datasets load from local JSON/CSV/JSONL (`add_goldens_from_json_file()` etc.), save locally via `save_as(file_type="json"|"csv", directory=...)`, or `push()`/`pull()` against Confident AI. At eval time you generate outputs per golden, convert to test cases, and run.
- **Execution surfaces.** Programmatic `evaluate(test_cases, metrics)` for scripts, and pytest-style `assert_test(test_case, metrics)` executed via `deepeval test run test_file.py` for CI ([getting started](https://deepeval.com/docs/getting-started)). There is also an `evals_iterator()` loop for dataset-driven runs with tracing.
- **Async and concurrency.** `evaluate()` takes an [`AsyncConfig`](https://deepeval.com/docs/evaluation-flags-and-configs) with `run_async` (default True), `max_concurrent` (default 20), and `throttle_value` for rate-limit backoff; each metric also has its own `async_mode` flag. The CLI mirrors this with `-n` parallel processes and `-c` cache flags.

## Metric library depth

All metrics share a common contract â€” score in 0â€“1, configurable `threshold` (default 0.5), `strict_mode`, `verbose_mode`, and a generated reason string ([metrics introduction](https://deepeval.com/docs/metrics-introduction)).

- **G-Eval.** [`GEval`](https://deepeval.com/docs/metrics-llm-evals) builds a custom LLM-judge metric from `name`, `criteria` (or explicit `evaluation_steps`), and `evaluation_params`; optional `rubric` and `evaluation_template`. It generates chain-of-thought evaluation steps, scores 1â€“10, and normalizes using token probabilities to reduce judge bias.
- **DAG metrics.** [`DAGMetric`](https://deepeval.com/docs/metrics-dag) evaluates via a `DeepAcyclicGraph` of `TaskNode`, `BinaryJudgementNode`, `NonBinaryJudgementNode`, and leaf `VerdictNode` (score 0â€“10 or a child metric like GEval). Each judgement node answers one narrow question, so the score comes from the deterministic path taken rather than the LLM picking a number.
- **Agentic metrics.** [`TaskCompletionMetric`](https://deepeval.com/docs/metrics-task-completion) scores AlignmentScore(task, outcome) extracted from the agent's full trace â€” it *requires* `@observe()` tracing, not a plain test case. [`ToolCorrectnessMetric`](https://deepeval.com/docs/metrics-tool-correctness) is deterministic (correctly-used tools / total tools called) over `tools_called` vs `expected_tools`, with `evaluation_params` (`ToolCallParams.INPUT_PARAMETERS`, `OUTPUT`) and `should_exact_match` to tighten matching; passing `available_tools` adds an optional LLM optimality judgement. The category also includes Argument Correctness, Step Efficiency, Plan Adherence, and Plan Quality ([metrics introduction](https://deepeval.com/docs/metrics-introduction)).
- **RAG metrics.** Contextual Relevancy / Precision / Recall (retriever side) and Answer Relevancy / Faithfulness (generator side), all requiring `retrieval_context` ([metrics introduction](https://deepeval.com/docs/metrics-introduction)).
- **Safety metrics.** Bias, Toxicity, Non-Advice, Misuse, PIILeakage, and Role Violation; deeper red-teaming lives in the separate DeepTeam package ([metrics introduction](https://deepeval.com/docs/metrics-introduction)).
- **Conversational metrics.** Knowledge Retention, Role Adherence, Conversation Completeness, Conversation Relevancy, plus Conversational G-Eval/DAG variants ([metrics introduction](https://deepeval.com/docs/metrics-introduction)).
- **Custom metrics.** Fully self-coded metrics subclass `BaseMetric`; there are also Hallucination, Summarization, JsonCorrectness, and a Ragas wrapper.

## Judge model routing

Metrics default to OpenAI via `OPENAI_API_KEY`, but every metric takes a `model` parameter and DeepEval ships provider adapters ([metrics introduction](https://deepeval.com/docs/metrics-introduction)):

- **OpenAI-compatible endpoints (the KAOS/LiteLLM case).** `deepeval set-local-model --model=<name> --base-url="http://host:port/v1/" --save` points all metrics at any OpenAI-compatible server, with a placeholder API key accepted for unauthenticated endpoints ([LM Studio](https://deepeval.com/integrations/models/lmstudio), [vLLM](https://deepeval.com/integrations/models/vllm) integration pages document the same command). There is also a dedicated [LiteLLM integration](https://deepeval.com/integrations/models/litellm): `deepeval set-litellm --model=... --base-url=...` on the CLI, or the `LiteLLMModel(model, api_key, base_url, temperature, ...)` Python class passed to `metric(model=...)`. Friction vs native OpenAI is low for routing itself.
- **Fully custom judges.** Subclass [`DeepEvalBaseLLM`](https://deepeval.com/guides/guides-using-custom-llms) implementing `load_model()`, `generate()`, `a_generate()`, and `get_model_name()`. The real friction is structured output: metrics pass a Pydantic `schema` argument, and if the model can't reliably emit valid JSON (docs recommend `instructor` or `lm-format-enforcer`) "metrics that require structured output will fail with a confusing `AttributeError`" â€” a practical concern when routing judges to smaller self-hosted models through LiteLLM.
- Config from `deepeval set-*` persists in a local dotfile (the `--save` flag), so KAOS could bake judge routing into the eval job image via env/CLI at container start.

## Library-only operation (no Confident AI)

- **Works offline:** all metric execution, `evaluate()`, `assert_test`/`deepeval test run`, local dataset files (JSON/CSV/JSONL load, `save_as()`), disk caching (`CacheConfig(use_cache, write_cache)`), and local result persistence â€” `DisplayConfig(results_folder=...)` writes timestamped JSON test-run files ([flags and configs](https://deepeval.com/docs/evaluation-flags-and-configs)). The docs are explicit that Confident AI is optional and recommended only for "shared reports, regression tracking, observability, or production monitoring" ([getting started](https://deepeval.com/docs/getting-started)).
- **Requires Confident AI (`CONFIDENT_API_KEY` / `deepeval login`):** dataset `push()`/`pull()` by alias, shareable web reports and regression comparison UI, trace/span dashboards, online production evals, and prompt management. There is **no self-hosted UI**: the results/reporting path without the SaaS is the JSON files in `results_folder` plus terminal output â€” i.e., KAOS would own aggregation and rendering itself.
- **Phones home by default:** anonymous PostHog usage telemetry (event names, metric names, anonymous UUID, public IP), disabled with `DEEPEVAL_TELEMETRY_OPT_OUT=1`; Sentry error reporting is opt-in via `ERROR_REPORTING=1` ([data privacy](https://deepeval.com/docs/data-privacy)). However, [issue #2497](https://github.com/confident-ai/deepeval/issues/2497) reports that v3.7.7 set the *global* OTel `TracerProvider` at import time and exported spans to a vendor OTLP endpoint (`otlp.nr-data.net`) with an embedded key, plus Sentry profiling â€” open at the time of writing and a hard flag for a self-hosted, network-policied cluster; pin versions and verify egress.

## Pydantic AI integration

The [Pydantic AI integration](https://deepeval.com/integrations/frameworks/pydanticai) is primarily a *tracing* bridge: `DeepEvalInstrumentationSettings()` passed to the agent's `instrument` parameter auto-instruments every `Agent` run so agent, LLM, and tool calls become DeepEval spans. Metrics attach to those spans via `next_agent_span(metrics=[...])` / `next_llm_span(metrics=[...])`, traces are annotated with `update_current_trace()` / `update_current_span()`, and dataset runs use `evals_iterator()` or `assert_test` in CI. The instrumentation and metric scoring work without Confident AI, but visualizing the resulting traces requires the platform â€” locally you get scores, not a trace UI.

## OTel surface

"Ingests OTel spans" means ingestion *into Confident AI*, not into the library: the platform accepts OTLP-over-HTTP at `https://otel.confident-ai.com` (no gRPC) authenticated with `x-confident-api-key`, mapping `gen_ai.*` semantic conventions plus a custom `confident.*` namespace onto its span model, and DeepEval ships a `ConfidentSpanExporter` that exports traces to the same SaaS observatory ([Confident AI OpenTelemetry docs](https://www.confident-ai.com/docs/integrations/opentelemetry)). There is no documented path to feed KAOS-collected OTel spans into offline DeepEval metrics; conversely, agentic metrics like TaskCompletion depend on DeepEval's own `@observe` tracing, which is a parallel tracing stack to KAOS's OTel pipeline rather than a consumer of it.

## Maturity

- **License and traction:** Apache-2.0, ~16.9k stars, ~9.8k commits, 57 releases with the latest (v4.1.0) published July 2026 â€” very active cadence ([repo](https://github.com/confident-ai/deepeval)).
- **Dependency weight:** heavier than a pure metric library â€” pulls OpenTelemetry SDK/exporters, PostHog and Sentry telemetry clients, an optional `litellm` extra, and pytest integration machinery; the import-time side effects in [#2497](https://github.com/confident-ai/deepeval/issues/2497) show that weight is not inert.
- **Governance:** single-vendor project whose roadmap tracks the Confident AI product; docs consistently funnel dataset management, reporting, and tracing toward the platform.

## How it would embed in KAOS

The strong fit is **supplementary metric library inside a KAOS-owned harness**: keep Pydantic Evals (or a thin KAOS runner) as the harness, and wrap selected DeepEval metrics â€” GEval, DAGMetric, ToolCorrectnessMetric, the RAG and safety sets â€” as evaluators over KAOS's own case/dataset format, constructing `LLMTestCase` objects (mapping KAOS tool-call records to `ToolCall`) and calling `metric.measure()`/`a_measure()` directly. Judge routing goes through the existing LiteLLM proxy via `LiteLLMModel` or `deepeval set-local-model` with the proxy base URL, keeping all judge traffic in-cluster. Results stay self-hosted by treating metric scores/reasons as KAOS data â€” emitted onto KAOS's OTel pipeline as span attributes/events â€” and ignoring DeepEval's run persistence entirely; `DEEPEVAL_TELEMETRY_OPT_OUT=1` is set in the eval job image. Using DeepEval **as the harness itself** (`evaluate()`, `deepeval test run`, `EvaluationDataset`) is workable offline but leaves reporting as raw JSON files, duplicates KAOS's tracing with `@observe`, and forfeits the trace-dependent metrics' main benefit without the SaaS UI â€” so it only makes sense if KAOS wanted DeepEval's pytest CI ergonomics more than its own orchestration.

**Risks / gaps**

- **SaaS gravity:** shared reports, regression UI, dataset hosting, trace visualization, and OTel ingestion are all Confident AI-only; there is no self-hosted server or UI, so the self-hosted result path is JSON files KAOS must aggregate itself ([getting started](https://deepeval.com/docs/getting-started), [flags and configs](https://deepeval.com/docs/evaluation-flags-and-configs)).
- **Egress and import-time side effects:** default PostHog telemetry plus the reported global-TracerProvider/OTLP exfiltration in [#2497](https://github.com/confident-ai/deepeval/issues/2497) demand version pinning, `DEEPEVAL_TELEMETRY_OPT_OUT=1`, and NetworkPolicy verification before cluster use â€” and importing DeepEval could clash with KAOS's own OTel `TracerProvider`.
- **Parallel tracing stack:** trace-dependent agentic metrics (TaskCompletion, Step Efficiency, Plan Adherence) require DeepEval's `@observe`/instrumentation rather than consuming KAOS OTel spans, so KAOS mostly gets the test-case-level metrics ([task completion](https://deepeval.com/docs/metrics-task-completion)).
- **Judge structured-output fragility:** metrics need schema-conformant JSON from the judge; smaller self-hosted models behind LiteLLM may fail with opaque errors unless wrapped with `instructor`/`lm-format-enforcer` ([custom LLM guide](https://deepeval.com/guides/guides-using-custom-llms)).
- **Single-vendor coupling and churn:** fast-moving APIs (v3â†’v4 within a year), roadmap driven by the Confident AI product, and heavier transitive dependencies than a metrics-only library warrant an adapter layer isolating KAOS from DeepEval's surface.
