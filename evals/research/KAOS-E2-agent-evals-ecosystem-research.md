# KAOS-E2 — Agent evaluation ecosystem research

This document surveys the LLM/agent evaluation ("evals") ecosystem as of 2025-2026, tool-agnostic and concepts-first. It establishes the shared vocabulary — taxonomy, evaluator kinds, dataset management, lifecycle, online practices, multi-agent concerns, and the benchmark landscape — that later KAOS eval design documents build on.

## 1. Evaluation taxonomy

### Offline vs online

The ecosystem has converged on a two-sided split. Offline evaluation runs a system-under-test against a fixed dataset of known cases (a "golden set") in a controlled environment, typically in CI or before a deploy, and its value is reproducibility: the same cases, scored the same way, catch regressions when a prompt, model, or tool configuration changes ([Braintrust](https://www.braintrust.dev/articles/llm-evaluation-guide), [qaskills.sh](https://qaskills.sh/blog/offline-vs-online-llm-evaluation-2026)). Online evaluation scores real production traffic — usually asynchronously, on traces rather than on curated inputs — and its value is coverage: it catches drift, novel inputs, frustrated users, and failure modes that no offline suite anticipated ([Morph](https://www.morphllm.com/ai-agent-evaluation), [Galtea](https://galtea.ai/blog/llm-evaluation-complete-guide)). The two are complementary, not alternatives: the standard practice is offline evals as pre-deploy gates plus continuous online evals on sampled production traces, with online failures flowing back into the offline dataset ([MLflow](https://mlflow.org/top-5-agent-evaluation-frameworks/), [Langfuse](https://langfuse.com/guides/cookbook/example_pydantic_ai_mcp_agent_evaluation)).

### End-to-end vs stepwise/trajectory

For single-shot LLM calls, evaluating the final output is enough. For agents, it is not: an agent produces a trajectory — a sequence of model calls, tool calls, arguments, and intermediate results — and a correct final answer can hide a broken process (say, 20 steps where 5 would do, or a policy-violating intermediate call) ([Morph](https://www.morphllm.com/ai-agent-evaluation), [Confident AI](https://www.confident-ai.com/blog/llm-agent-evaluation-complete-guide)). Trajectory evaluation therefore decomposes into recognizable sub-metrics:

- Tool selection correctness — did the agent invoke the right tool for the step, out of the tools available to it ([Confident AI](https://www.confident-ai.com/blog/llm-agent-evaluation-complete-guide))?
- Argument/parameter correctness — were the tool call arguments valid and semantically right, the core competency measured by function-calling benchmarks like BFCL ([Spheron](https://www.spheron.network/blog/tool-calling-benchmarks-bfcl-tau-bench-latency-optimization/))?
- Path efficiency and unnecessary steps — number of steps, redundant or looping calls, backtracking; often scored against a reference trajectory (exact match, in-order match, any-order match) or as a step-count ratio ([Vinod Rane](https://medium.com/@vinodkrane/chapter-8-agent-evaluation-for-llms-how-to-test-tools-trajectories-and-llm-as-judge-788f6f3e0d52), [emergentmind](https://www.emergentmind.com/topics/llm-agent-evaluation-frameworks)).
- Task success — the end-to-end outcome check, independent of path; the survey literature describes the field's evolution as moving from static, one-shot, text-centric benchmarks toward dynamic, process-sensitive, system-level evaluation that scores both ([KDD 2025 tutorial](https://sap-samples.github.io/llm-agents-eval-tutorial/2025_KDD_Evaluation_and_Benchmarking_of_LLM_Agents.pdf), [arXiv 2411.13768](https://arxiv.org/pdf/2411.13768)).

The practical framing: end-to-end metrics tell you *whether* the agent failed; trajectory metrics tell you *where* — which is what makes them actionable for debugging and regression triage.

### Single-turn vs multi-turn

Single-turn evaluation scores one input/output exchange; multi-turn (session/conversation) evaluation scores an entire dialogue, where quality depends on context retention, follow-up handling, and recovery from earlier mistakes ([ACM survey on multi-turn agent evaluation](https://dl.acm.org/doi/10.1145/3793671), [arXiv 2503.22458](https://arxiv.org/abs/2503.22458)). Multi-turn evaluation is materially harder and more expensive: annotating a whole dialogue takes more effort than a single response, evaluators often use moving-window approaches to score segments incrementally, and offline multi-turn cases need either scripted user turns or a user simulator (as pioneered by tau-bench) to drive the conversation deterministically ([emergentmind](https://www.emergentmind.com/topics/multi-turn-evaluation-protocol), [benchmarkingagents.com](https://benchmarkingagents.com/tau-bench/)). A useful distinction is turn-level metrics (quality of one response given conversation history) vs session-level metrics (goal completion, conversation efficiency, overall coherence) ([Data Science XAI](https://datasciencexai.substack.com/p/beyond-single-turn-evaluating-multi)).

### Granularity and purpose

Cutting across the above, evaluations attach at different granularities: span level (one model or tool call), trace level (one agent task end to end), and session level (a multi-turn conversation or long-running interaction), and mature tooling lets the same evaluator vocabulary target any of the three ([Langfuse](https://langfuse.com/guides/cookbook/example_pydantic_ai_mcp_agent_evaluation), [Confident AI](https://www.confident-ai.com/blog/llm-agent-evaluation-complete-guide)). It is also worth distinguishing purpose: capability evaluation asks "how good is this system at the task?" (benchmark-style, informative during development), while regression evaluation asks "did this specific change make it worse?" (gate-style, comparative against a baseline) — the same datasets and evaluators serve both, but thresholds and reporting differ ([Braintrust](https://www.braintrust.dev/articles/llm-evaluation-guide), [arXiv 2411.13768](https://arxiv.org/pdf/2411.13768)).

## 2. Evaluator kinds

Evaluators (also called scorers, graders, or metrics) fall into a small number of well-established kinds, ordered roughly by cost and by determinism.

### Deterministic / code assertions

Plain code that checks a property of the output or trajectory: exact/substring match, regex, JSON schema validity, "the answer contains the expected order ID", "tool X was called before tool Y", latency under N seconds. These are cheap, fast, perfectly reproducible, and should be the first choice wherever the property is objectively checkable ([Braintrust](https://www.braintrust.dev/articles/llm-evaluation-guide)). For agents specifically, many trajectory checks (tool name match, argument schema validity, step count bounds) are deterministic even when the final answer needs a fuzzier judge.

### Statistical / heuristic metrics

Classic NLP similarity scores (BLEU, ROUGE, embedding cosine similarity) and heuristics (length, readability, keyword coverage). They are reference-based, cheap, and language-model-free, but correlate weakly with human judgment on open-ended generation, so the 2025-era consensus treats them as supporting signals rather than primary quality gates ([Galtea](https://galtea.ai/blog/llm-evaluation-complete-guide), [arXiv 2506.13023](https://arxiv.org/pdf/2506.13023)).

### LLM-as-judge

An LLM scores an output (or full trajectory) against a rubric. This is the workhorse for subjective qualities — helpfulness, faithfulness, tone, instruction adherence — because it scales where human review cannot. The key design points:

- Rubric design — decompose "quality" into narrow, explicitly defined criteria; instance-level rubrics that reduce evaluation to binary checks per criterion align better with humans than open-ended 1-10 scoring ([ACM survey](https://dl.acm.org/doi/10.1145/3793671), [emergentmind](https://www.emergentmind.com/topics/llm-as-a-judge-evaluations)).
- Pointwise vs pairwise — pointwise assigns an absolute score to one output; pairwise picks the better of two. Pairwise comparison consistently aligns better with human annotators and reduces anchoring effects seen in Likert-style ratings, which is why it dominates model comparison and A/B experiments, while pointwise remains necessary for threshold gating on a single system ([arXiv 2508.00454](https://arxiv.org/html/2508.00454v4), [emergentmind](https://www.emergentmind.com/topics/multi-turn-evaluation-protocol)).
- Known biases — LLM judges exhibit position bias (verdicts flip in 10-30% of pairwise comparisons when candidate order is swapped), verbosity bias (preferring longer, more fluent outputs regardless of substance), and self-preference/family bias (scoring their own or same-provider outputs roughly 10% higher, correlated with lower perplexity of the judged text) ([arXiv 2410.02736 "Justice or Prejudice?"](https://arxiv.org/pdf/2410.02736), [arXiv 2410.21819 self-preference](https://arxiv.org/pdf/2410.21819), [Adaline](https://www.adaline.ai/blog/llm-as-a-judge-reliability-bias)).
- Mitigations and calibration — randomize or swap presentation order and average, control for length, use a judge from a different model family than the system under test, and — most importantly — calibrate the judge against a small human-labeled set and track judge-human agreement as its own metric; an uncalibrated judge is an unvalidated measurement instrument ([Vadim's blog](https://vadim.blog/llm-as-judge/), [Adaline](https://www.adaline.ai/blog/llm-as-a-judge-reliability-bias)).
- Judge versioning — the judge prompt and judge model are part of the measurement instrument; changing either invalidates trend comparisons unless the baseline is re-run and the boundary is tagged ([FutureAGI](https://futureagi.com/glossary/llm-regression-testing/)).

Two further operational distinctions matter for judges. First, reference-based vs reference-free: a reference-based judge compares the output against an expected answer from the dataset (only possible offline), while a reference-free judge scores against a rubric using only the input, output, and retrieved/tool context — which is what makes LLM-as-judge the primary quality signal in production, where no reference exists ([qaskills.sh](https://qaskills.sh/blog/offline-vs-online-llm-evaluation-2026)). Second, output-level vs trajectory-level judging: judges can be given the full agent trajectory (tool calls, arguments, intermediate results) and asked process questions — "did the agent use the retrieved information, or ignore it?" — which is increasingly standard for agent evaluation even though it raises judge context length and cost ([Confident AI](https://www.confident-ai.com/blog/llm-agent-evaluation-complete-guide), [Vinod Rane](https://medium.com/@vinodkrane/chapter-8-agent-evaluation-for-llms-how-to-test-tools-trajectories-and-llm-as-judge-788f6f3e0d52)).

### Human annotation loops

Humans remain the ground truth: they label the calibration sets that judges are validated against, review low-confidence or disputed outputs routed to annotation queues, and provide the thumbs-up/down feedback that seeds datasets ([Braintrust](https://www.braintrust.dev/articles/llm-evaluation-guide), [ACM survey](https://dl.acm.org/doi/10.1145/3793671)). The economics dictate the pattern: humans annotate a small, high-value slice (calibration sets, escalations, disagreements between judges), and automated evaluators extrapolate from there. Binary choices and pairwise comparisons yield better inter-annotator agreement than fine-grained rating scales, so annotation UIs trend toward those formats ([emergentmind](https://www.emergentmind.com/topics/multi-turn-evaluation-protocol)).

## 3. Dataset and golden-set management

A golden dataset is a curated, versioned collection of test cases — each with inputs, expected behavior, and metadata — that serves as the trusted reference for comparing runs over time ([Klu](https://klu.ai/glossary/golden-dataset), [Maxim](https://www.getmaxim.ai/articles/building-a-golden-dataset-for-ai-evaluation-a-step-by-step-guide/)). The recurring schema across the ecosystem is a case (often called a "golden" or "example") of roughly: input (the user message or task, plus any context the system needs), expected output (a reference answer, expected tool-call sequence, or expected properties — which may be absent for reference-free judge metrics), and metadata (tags, difficulty, source, category) used for slicing results ([DeepEval datasets](https://deepeval.com/docs/evaluation-datasets), [Confident AI](https://documentation.confident-ai.com/concepts/datasets)). For agents, expected output frequently includes a reference trajectory (expected tools and argument constraints) rather than only a final answer.

Practices that the ecosystem treats as table stakes:

- Small but ruthlessly curated — a useful golden set is task-specific and typically in the hundreds to low thousands of cases, hand-checked rather than large and noisy ([qaskills.sh](https://qaskills.sh/blog/golden-dataset-llm-evaluation-guide)).
- Versioned like code — every dataset change is versioned with an audit trail, so a score change can always be attributed to either a system change or a dataset change, never ambiguously to both ([Maxim](https://www.getmaxim.ai/articles/building-a-golden-dataset-for-ai-evaluation-a-step-by-step-guide/), [FutureAGI](https://futureagi.com/glossary/llm-regression-testing/)).
- Synthetic generation to bootstrap — when real data is scarce, an LLM generates seed inputs and then "evolves" them into more complex and realistic variants, for both single-turn and multi-turn cases; synthetic goldens still need human review before joining the trusted set ([DeepEval synthesizer](https://deepeval.com/docs/golden-synthesizer)).
- Production traces as the best source — the highest-value cases come from production failures: thumbs-down feedback, escalations, abandoned sessions, and flagged low-confidence outputs get converted into dataset cases, closing the online-to-offline loop ([qaskills.sh](https://qaskills.sh/blog/golden-dataset-llm-evaluation-guide), [Braintrust](https://www.braintrust.dev/articles/llm-evaluation-guide)). This loop is also the main answer to the widely reported complaint that static offline suites decay and under-deliver ([Morph](https://www.morphllm.com/ai-agent-evaluation)).

## 4. Regression gating and lifecycle

The ecosystem has settled on treating evals as the LLM equivalent of a test suite, wired into the change lifecycle:

- Evals in CI — every change to a prompt, model version, tool definition, or retrieval configuration triggers an eval run against the golden dataset; a PR that regresses quality past threshold does not merge ([promptfoo/CI walkthrough](https://medium.com/@alexrodriguesj/testing-llm-prompts-like-code-regression-evals-in-ci-cd-with-promptfoo-5242b4dcb9be), [Latitude](https://latitude.so/blog/ultimate-ci-cd-llm-evaluation-guide)).
- Pass/fail thresholds — eval gates block a change if scores fall below defined thresholds; thresholds must reflect minimum acceptable quality, since gates that fail constantly provide no signal and gates that never fail catch nothing ([Galtea](https://galtea.ai/blog/automated-llm-evaluation-building-a-ci-cd-quality-gate-that-actually-runs), [FutureAGI](https://futureagi.com/glossary/llm-regression-testing/)). Because LLM outputs are non-deterministic, gates compare aggregate scores (mean, pass-rate over N cases, sometimes with repeat runs) rather than exact outputs.
- Experiments, not just gates — each eval run is recorded as an experiment (system config + dataset version + evaluator versions + per-case scores), enabling side-by-side comparison of two candidates and drill-down into exactly which cases regressed and by how much ([Braintrust eval-driven development](https://www.braintrust.dev/articles/eval-driven-development)). Pairwise/A-B comparison between a baseline experiment and a candidate experiment is the standard mechanism for prompt and model change management ([Traceloop](https://www.traceloop.com/blog/automated-prompt-regression-testing-with-llm-as-a-judge-and-ci-cd)).
- Everything versioned together — meaningful trend lines require pinning the tuple (dataset version, evaluator/judge version, system version); changing any one without tagging the boundary turns the trend into noise ([FutureAGI](https://futureagi.com/glossary/llm-regression-testing/)).

- Handling non-determinism — since a single run can pass or fail by luck, robust gates run each case multiple times and gate on pass-rate or score distributions, and treat flaky cases (inconsistent across repeats at the same config) as a distinct category to investigate rather than silently average away ([Galtea](https://galtea.ai/blog/automated-llm-evaluation-building-a-ci-cd-quality-gate-that-actually-runs), [Latitude](https://latitude.so/blog/ultimate-ci-cd-llm-evaluation-guide)).

- Tiered suites for cost — full golden-set runs with LLM judges are slow and expensive, so teams commonly tier them: fast deterministic checks on every commit, a representative subset plus judges on PRs, and the full suite on merge or nightly ([Braintrust](https://www.braintrust.dev/articles/best-ai-evals-tools-cicd-2025)).

## 5. Production / online practices

### Sampling and asynchrony

Online evaluators run asynchronously against production traces so they never add user-facing latency, and they sample rather than score everything — a common baseline is scoring 5-10% of traffic continuously, with rules that up-sample interesting traffic (errors, negative feedback, specific routes or tenants, high-cost sessions) to 100% ([qaskills.sh](https://qaskills.sh/blog/offline-vs-online-llm-evaluation-2026), [TrueFoundry](https://www.truefoundry.com/blog/online-llm-evaluation-gateway)). Because production has no reference answers, online evaluation leans on reference-free signals: LLM-as-judge against a rubric, heuristic checks, and explicit/implicit user feedback ([qaskills.sh](https://qaskills.sh/blog/offline-vs-online-llm-evaluation-2026)).

Sampling strategy is itself a design decision with recognizable options: uniform random sampling for unbiased trend lines, stratified sampling to guarantee coverage of low-traffic but important segments, and event-triggered scoring (error status, negative feedback, guardrail hits, unusually long or expensive sessions) to concentrate judge budget where failures are likeliest ([TrueFoundry](https://www.truefoundry.com/blog/online-llm-evaluation-gateway), [Galileo](https://galileo.ai/blog/best-llm-output-drift-monitoring-platforms)). Whatever the strategy, the sampled-and-scored traces plus their scores are the raw material for the trace-to-dataset loop from section 3.

### Guardrails vs evals

These are distinct mechanisms with different latency budgets, and the boundary matters architecturally. A guardrail runs inline, synchronously, in milliseconds, and blocks a specific clear-cut failure (PII leak, prompt injection, invalid JSON, profanity) using rules, schemas, or lightweight classifiers. An eval runs after the fact, asynchronously, may take seconds and use an LLM judge, and measures quality rather than blocking anything ([Laika](https://laikatest.com/blogs/understanding-guardrails-moderation-evals), [Galtea](https://galtea.ai/blog/llm-evaluation-complete-guide)). A practical corollary: a repeatedly failing online eval is a candidate for a new guardrail, but evals themselves should not sit on the request path.

### Drift and first-class cost/latency

Online evaluation is the mechanism that catches drift: input distributions, user behavior, and upstream model behavior all shift over time, so a system that scored 92% offline months ago may be quietly handling a different query mix now ([qaskills.sh](https://qaskills.sh/blog/offline-vs-online-llm-evaluation-2026), [Galileo](https://galileo.ai/blog/best-llm-output-drift-monitoring-platforms)). For agents, cost and latency are first-class eval metrics, not just ops metrics — token usage, dollar cost per task, wall-clock duration, and step counts are scored alongside quality, because an agent that gets the right answer at 10x the token budget is a regression ([Confident AI](https://www.confident-ai.com/blog/llm-agent-evaluation-complete-guide), [Morph](https://www.morphllm.com/ai-agent-evaluation)).

### OpenTelemetry GenAI semantic conventions

The OTel GenAI conventions are the emerging lingua franca for the traces that online (and increasingly offline) evaluation consumes. Current state:

- The conventions define agent invocation spans (`invoke_agent`), inference spans (`chat`), and tool execution spans (`execute_tool`), with attributes like `gen_ai.request.model`, `gen_ai.usage.input_tokens`/`output_tokens`, `gen_ai.response.finish_reasons`, and opt-in content capture via `gen_ai.input.messages`/`gen_ai.output.messages`, plus the metrics `gen_ai.client.operation.duration` and `gen_ai.client.token.usage` ([OpenTelemetry blog](https://opentelemetry.io/blog/2026/genai-observability/)).
- Stability caveat: as of 2026 the GenAI (and MCP) conventions remain in Development status — attribute names can still change without a major version bump — though vendor and framework adoption is already broad ([Greptime](https://greptime.com/blogs/2026-05-09-opentelemetry-genai-semantic-conventions), [Fiddler](https://www.fiddler.ai/blog/opentelemetry-ai-observability-guide)); the conventions now live in a dedicated [semantic-conventions-genai repository](https://github.com/open-telemetry/semantic-conventions-genai) under the GenAI SIG.
- Evaluation results have a standardized shape: the `gen_ai.evaluation.result` event (merged into semconv v1.39, Aug 2025) carries `gen_ai.evaluation.name`, `gen_ai.evaluation.score.value`, `gen_ai.evaluation.score.label`, and `gen_ai.evaluation.explanation`, and SHOULD be parented to the GenAI operation span being evaluated (or correlated via `gen_ai.response.id`) ([gen-ai events spec](https://github.com/open-telemetry/semantic-conventions-genai/blob/main/docs/gen-ai/gen-ai-events.md), [Datadog on OTel GenAI adoption](https://www.datadoghq.com/blog/llm-otel-semantic-convention/)). This means an eval system can both *read* standardized traces as its input and *write* standardized evaluation events back onto them, keeping scores queryable next to the spans they judge.

Trace-based evaluation builds directly on this: the span tree *is* the trajectory. Tool selection, argument correctness, step counts, token cost, and latency are all recoverable from a conventions-compliant trace without any bespoke logging, which is why trace-first eval architectures have become the default for agent frameworks that emit OTel natively ([Zylos](https://zylos.ai/research/2026-02-28-opentelemetry-ai-agent-observability/), [Langfuse](https://langfuse.com/guides/cookbook/example_pydantic_ai_mcp_agent_evaluation)).

## 6. Multi-agent and delegation evaluation

Multi-agent systems add a layer the single-agent taxonomy does not cover: the unit under evaluation can be the orchestrator (did it decompose the task sensibly, route to the right specialist, and synthesize sub-results correctly?), an individual sub-agent (did it complete its delegated sub-task, judged against the sub-task's own success criteria rather than the end user's goal?), or the system as a whole ([arXiv orchestration survey](https://arxiv.org/html/2601.13671v1), [KDD 2025 tutorial](https://sap-samples.github.io/llm-agents-eval-tutorial/2025_KDD_Evaluation_and_Benchmarking_of_LLM_Agents.pdf)).

The hard problem is credit assignment: when an end-to-end task fails, determining which agent (or which handoff between agents) caused the failure. In practice this is approached structurally — the delegation trace forms a tree of agent invocations, and stepwise evaluation is applied per node so that failures localize to a sub-span rather than being attributed only to the top-level outcome; this is the multi-agent generalization of "trajectory metrics tell you where" ([emergentmind](https://www.emergentmind.com/topics/llm-agent-evaluation-frameworks), [arXiv 2503.22458](https://arxiv.org/abs/2503.22458)).

Delegation protocols make this tractable. A2A-style delegation — where an orchestrator delegates a task to a remote agent with structured task, message, and artifact objects — produces explicit, inspectable handoff boundaries: the delegated task specification and the returned artifact can each be evaluated independently ("was the delegation well-specified?" vs "did the sub-agent fulfill the specification?") ([IBM on A2A](https://www.ibm.com/think/topics/agent2agent-protocol), [MindStudio](https://www.mindstudio.ai/blog/what-is-a2a-agent-to-agent-protocol)). When delegation flows through OTel-instrumented spans (nested `invoke_agent` spans), the same trace-based evaluation machinery from section 5 applies recursively, which is the practical convergence point of multi-agent evaluation and observability ([Zylos](https://zylos.ai/research/2026-02-28-opentelemetry-ai-agent-observability/)).

## 7. Agent benchmark landscape (as vocabulary)

Public benchmarks matter here mainly as shared vocabulary for evaluation *design patterns*, not as adoption candidates:

- [tau-bench / tau2-bench](https://benchmarkingagents.com/tau-bench/) — customer-service agents in retail/airline/telecom domains; notable for its user simulator driving multi-turn dialogue, explicit per-domain policies the agent must obey, and stateful environments where actions have lasting consequences — the reference pattern for policy-adherence and simulated-user evaluation.
- [SWE-bench](https://www.layer3labs.io/guides/ai-agent-benchmarks) — coding agents resolving real GitHub issues, scored by held-out unit tests; the reference pattern for outcome-verified evaluation via executable checks.
- [GAIA](https://awesomeagents.ai/leaderboards/agentic-ai-benchmarks-leaderboard/) — real-world assistant questions requiring reasoning, web browsing, and tool use, with unambiguous factual answers; the pattern for tasks easy for humans, hard for agents, cheap to score.
- [AgentBench](https://prefactor.tech/learn/agent-benchmarks) — a multi-environment suite (OS, database, web, games) testing general agentic ability across interactive environments.
- [BFCL (Berkeley Function-Calling Leaderboard)](https://www.spheron.network/blog/tool-calling-benchmarks-bfcl-tau-bench-latency-optimization/) — the standard for tool/function-calling correctness: right function, valid and correct parameters, including multi-turn and stateful variants in v4; the vocabulary source for "tool selection" and "argument correctness" metrics.

The common thread: modern benchmarks moved from static Q&A to interactive, stateful environments with verifiable outcomes — exactly the offline-eval properties (deterministic environments, executable checks, simulated users) that in-house agent eval suites now emulate ([KDD 2025 tutorial](https://sap-samples.github.io/llm-agents-eval-tutorial/2025_KDD_Evaluation_and_Benchmarking_of_LLM_Agents.pdf)).

## Implications for KAOS

- KAOS is OTel-first, so trace-based evaluation is the natural foundation: the Pydantic AI runtime's spans already encode trajectories (agent, model, tool spans), and evaluators should consume those rather than a bespoke log format.
- Align span/attribute naming with the OTel GenAI conventions (`invoke_agent`, `execute_tool`, `gen_ai.*`) and emit eval scores as `gen_ai.evaluation.result` events parented to the evaluated span — while tracking the conventions' Development status, since attribute names may still shift.
- The offline/online split maps cleanly to Kubernetes-native shapes: offline eval runs are batch, Job-like executions against a versioned dataset; online evaluation is a continuous, sampled consumer of production traces.
- An eval case schema needs agent-specific fields from day one: expected tool calls / trajectory constraints and metadata for slicing, not just input/expected-output pairs.
- Evaluator kinds should be a small closed set initially — deterministic assertions and LLM-as-judge cover most needs — with judge prompts and judge models versioned as part of the measurement instrument, and judge model family kept separate from the agent's ModelAPI to avoid self-preference bias.
- Dataset, evaluator, and agent versions must be pinned together per eval run (an "experiment") so score changes are attributable; this fits naturally with declarative, versioned CRD specs.
- The production-trace-to-dataset loop is the highest-leverage feature to design for: promoting a failing trace (or annotated span) into a dataset case is what keeps golden sets from decaying.
- Cost (tokens), latency, and step count should be first-class eval metrics alongside quality scores, derived from the same spans.
- Guardrails and evals are different subsystems: evals must stay off the request path (async, sampled), and any inline blocking behavior belongs to a separate guardrail mechanism, not the eval capability.
- Multi-agent/delegation evaluation should fall out of the trace model recursively: each nested agent-invocation span is evaluable against its delegated sub-task, giving structural credit assignment without new machinery.
