# P1 — `kaos-evals` library and runner

**Status**: Ready for execution (v1)
**Scope**: The first phase of [proposed-split](./proposed-split.md): the complete evaluation capability as a Python package — engine validation, the KAOS-owned contract, gate logic, the Pydantic Evals harness boundary, target adapters, the runner entrypoint, and OTel instrumentation — runnable locally end to end with no cluster. Realises [adr_0001](../adrs/adr_0001_eval-model-contract-and-harness-engine.md) fully and the non-Job decisions of [adr_0002](../adrs/adr_0002_execution-model-and-runner.md).

## Design plan: fit with the existing codebase

- **Package home**: a new top-level `kaos-evals/` directory in the KAOS repo, sibling to `kaos-memory/` and structured identically: `kaos_evals/` package, `tests/`, `pyproject.toml` (uv, hatchling, Apache-2.0, `requires-python >=3.12`), `Makefile` (`build`, `test`, `lint`, `format` targets; lint = `black --check` + `ty`, matching `kaos-memory/Makefile`), `README.md`, `Dockerfile` deferred to the control-plane phase.
- **Dependency layering (mirrors `kaos-memory/pyproject.toml`)**: core dependencies are `pydantic`, `pyyaml`, `opentelemetry-api` only — the contract and gates import no engine. The `[runner]` extra adds `pydantic-evals` (pinned exact version), `pydantic-ai` (for judge model objects and the local adapter), `httpx`, `opentelemetry-sdk`. The `[dev]` extra adds `pytest`, `pytest-asyncio`, `black`.
- **Module layout** (one concern per module, mirroring `kaos_memory`'s `contract.py`/`client.py`/`app.py` discipline):
  - `kaos_evals/contract.py` — `EvaluatorSpec`, `GateSpec`, `EvalCase`, `EvalSuite`, `RunSpec`, `TargetSpec`, `EvaluatorResult`, `CaseResult`, `GateEvaluation`, `RunResult`, `Provenance`, `Verdict` enum (`passed|failed|error`), `FailureKind` enum; suite YAML/JSON load + canonical serialization + `suite_hash()` (SHA-256 of canonical form).
  - `kaos_evals/gates.py` — pure verdict computation from case results + `GateSpec` (minPassRate, thresholds, criticalTags, budget ceilings), honouring the failure-kind taxonomy (case failure vs inconclusive evaluator vs run error).
  - `kaos_evals/harness.py` — the engine boundary: suite → `pydantic_evals.Dataset`/`Case`/evaluator instances; engine report → `RunResult`; the built-in evaluator kinds (`contains`, `equals`, `regex`, `is_json`, `max_duration`, `llm_judge`); judge model construction (`OpenAIChatModel` + `OpenAIProvider(base_url=...)`, explicit model required, fail-closed if unresolved).
  - `kaos_evals/targets.py` — the adapter interface (`async def __call__(case) -> TargetResponse` with output, duration, usage, trace ID, typed error) and the two adapters: `HttpTarget` (OpenAI-compatible chat endpoint, fresh session ID per case+repetition, KAOS-side per-case timeout via `asyncio.timeout`, W3C traceparent propagation and trace-ID capture) and `LocalTarget` (in-process ASGI transport against an `AgentServer` app or any OpenAI-compatible URL).
  - `kaos_evals/runner.py` — the CLI entrypoint (`python -m kaos_evals.runner` / console script `kaos-evals-runner`): args/env for suite path, target URL or mode, judge base-URL map, output paths; orchestrates load → hash → execute → gate → write `summary.json` + `report.json` → exit 0/1/2 for passed/failed/error.
  - `kaos_evals/telemetry.py` — OTel spans (`kaos.eval.run`, `kaos.eval.case`, `kaos.eval.evaluator`) and `gen_ai.evaluation.result` score events, following `pydantic-ai-server/pais/telemetry.py` conventions; no-op safe when no SDK is configured.
- **Testing pattern**: unit tests with fakes (a fake OpenAI-compatible endpoint via `httpx.MockTransport` / a minimal ASGI app, an injected `FunctionModel` for judges) mirroring `kaos-memory/tests/_fakes.py`; one cross-component test standing up a real `AgentServer` from `pydantic-ai-server` with `DEBUG_MOCK_RESPONSES` and running a suite against it through `LocalTarget`.
- **What this phase deliberately does not touch**: the operator, CRDs, Helm, kaos-cli, docs/examples, the runner Dockerfile — all later phases per [proposed-split](./proposed-split.md).

## Execution conventions (binding for this plan)

- Execute the numbered TODOs below strictly one by one, without skipping; do not start a task before the previous one is committed.
- Every task ends with the package's tests validated and passing (`uv run pytest tests/ -v` and `make lint` in `kaos-evals/`; plus any other suite the task touches) and a **comprehensive conventional commit** — succinct, functional, describing what was added with context, and **never referencing tasks, phases, or plan steps** in the commit message or branch name.
- All scratch work goes in `./tmp/` (gitignored), never `/tmp`; suppress noise with `./tmp/null`.
- When all tasks are done, write a gitignored `REPORT.md` (overwrite if present) at the repo root covering every requested task and its completion status; it is **not committed** — its contents are posted as a comment on the phase PR.
