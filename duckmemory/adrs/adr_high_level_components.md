# DuckMemory ADRs — high-level components

This document opens the architecture-decision phase for **DuckMemory**: a standalone, embedded DuckDB extension that serves as a full end-to-end agent memory engine directly inside a DuckDB database. The research stage ([DUCK-R1](../research/DUCK-R1-duckdb-extension-ecosystem.md), [DUCK-R2](../research/DUCK-R2-reference-architectures.md)) established the DuckDB extension ecosystem's capabilities and hard boundaries, and distilled the two reference architectures (the KAOS memory subsystem for the domain, the `agent_data_duckdb` extension for the implementation skeleton). This document decomposes the target into the **high-level components that each require an architecture decision record (ADR)**, so the design work can proceed component by component.

The intent here is deliberately *coarse*. The components below are bundled rather than finely segregated: each one groups a set of closely-coupled decisions that are best made together, so the ADR set stays small and coherent instead of fragmenting into many narrow records. Each component listed will become one ADR (occasionally a small number, if a decision later proves too large to bundle), and this document is the index that anchors them.

## Positioning and settled direction

DuckMemory is **fully standalone**: it defines its own memory model and has no dependency on KAOS, though the KAOS memory ADRs are a primary evidence base and any agent framework — KAOS included — is a potential consumer. Three directional choices are already settled and the relevant ADRs record and elaborate them rather than reopening them: the implementation is **Rust in a new dedicated repository** following the `agent_data_duckdb` skeleton; vector and keyword search **reuse the `vss` and `fts` core extensions** for the initial version (iterating after an MVP exists) rather than building a custom index, which [DUCK-R1](../research/DUCK-R1-duckdb-extension-ecosystem.md) shows is not reachable from the Rust C API in any case; and the embedding/LLM integration boundary is deliberately **not** settled — it is the dedicated decision of component 4, to be made with explicit trade-offs.

## How this relates to the research

The components are grounded in the capability constraints of [DUCK-R1](../research/DUCK-R1-duckdb-extension-ecosystem.md) (what a Rust C-API extension can and cannot do; the `vss`/`fts` caveats; the LLM-in-database precedents; community distribution) and the requirements distilled in [DUCK-R2](../research/DUCK-R2-reference-architectures.md) (the tiered model, cascade write, scope-pre-filtered retrieval, and degradation semantics proven out by KAOS memory; the build/test skeleton proven out by `agent_data_duckdb`). Where a DuckMemory ADR weighs a domain option in depth, the underlying KAOS memory research ([KAOS-R1](../../memory/research/KAOS-R1-memory-features-and-limitations.md) through [KAOS-R11](../../memory/research/KAOS-R11-short-term-memory-storage.md)) and ADRs ([memory adr_0001](../../memory/adrs/adr_0001_memory-model-and-lifecycle-operations.md) onwards) are the extended evidence base. ADRs decide; they do not re-litigate the research. Each ADR records what was chosen, why, and the consequences, citing the research rather than repeating it.

## Components requiring an ADR

The five components below cover the full target surface. They are listed in authoring order rather than strictly by dependency: the conceptual memory model is settled first, the SQL surface that exposes it follows, the storage and retrieval implementation beneath it comes next, the embedding/LLM boundary is decided once the surfaces it sits between are fixed, and the extension architecture and distribution close the set. Cross-cutting concerns that belong to no single component are listed separately afterwards.

### 1. Memory model and lifecycle operations

- **Scope.** The conceptual memory model DuckMemory commits to, independent of SQL syntax or storage: the tier taxonomy (a verbatim short-term window, a rolling session digest, long-term atomic facts — and whether profile, temporal, or procedural tiers are in scope or deferred); the metadata envelope every item carries (id, scope keys, event/ingestion time, provenance, importance, access stats, retention/pin policy); the scope/namespace model — the dimensions along which memories are partitioned (for example agent, user, session, and a shared level) and whether scope is a flat set of owner keys or a hierarchical path; the retrieval scoring model (hybrid recall fusing vector relevance, keyword relevance, recency with decay, and importance); consolidation semantics (when short-term turns fold into digest and facts, and what folding means); and forgetting semantics (decay, pinning, and scope-targeted hard erasure across all tiers).
- **Key decisions.** Which tiers ship in the initial version and their sequence; the envelope schema that keeps tiers interoperable; the scope model — where [DUCK-R2](./../research/DUCK-R2-reference-architectures.md) notes the KAOS finding that a small flat set of owner keys maps cleanly onto pre-filtered retrieval, while a standalone engine may favour a free-form namespace key so host frameworks can encode their own hierarchy; the scoring formula and its configurability; the consolidation trigger model (token budget and thresholds) in the abstract, with its *execution* placement deferred to component 2; and the forgetting primitives. A distinguishing question for an embedded engine: how much of the KAOS-style cascade (short-term → digest → facts) is core versus optional, given hosts range from a single local agent to a framework wanting only the long-term fact store.
- **Primary inputs.** [DUCK-R2](../research/DUCK-R2-reference-architectures.md) sections 1 and 3; [memory adr_0001](../../memory/adrs/adr_0001_memory-model-and-lifecycle-operations.md) and [memory adr_0003](../../memory/adrs/adr_0003_memory-interface-and-runtime-data-plane.md) for the proven tier and lifecycle semantics; [KAOS-R2](../../memory/research/KAOS-R2-memory-ecosystem-research.md) for the ecosystem taxonomy.
- **Dependencies.** The conceptual foundation. Defines what the SQL surface (component 2) must expose, what the storage schema (component 3) must hold, and the seams the embedding/LLM boundary (component 4) plugs into.

### 2. SQL interface and embedded execution model

- **Scope.** The entire user-facing surface — in an embedded extension the API *is* SQL: the function set (write/remember, recall/search, forget, consolidate, stats/inspect) and the form each takes (scalar function, table function, or macro); naming and namespacing conventions (function prefix, the schema DuckMemory's tables live in); the shape of recall results (a table function returning the assembled tiered context versus separate per-tier queries); configuration (extension settings, pragmas, and how per-store options like embedding dimensions are declared); initialization (how a database becomes a memory store — an explicit `memory_init()` versus lazy creation); transactional semantics of each operation; and — because a daemonless engine has no background executor ([DUCK-R2](../research/DUCK-R2-reference-architectures.md) section 1) — the **execution model for consolidation and extraction**: explicit host-invoked calls, opportunistic piggybacking on writes or recalls, or a documented host-side scheduling pattern.
- **Key decisions.** The minimal function set and its ergonomics for the primary consumer (an agent framework calling through a DuckDB client library); whether recall is one call returning the full context block or composable primitives the host assembles; the consolidation execution model — the single most consequential embedded-versus-service divergence, since everything KAOS ran in a background executor must here run inside somebody's SQL statement; idempotency and concurrency behaviour when multiple connections or processes open the same database file (DuckDB allows one writer); and what is guaranteed stable API versus internal schema.
- **Primary inputs.** [DUCK-R2](../research/DUCK-R2-reference-architectures.md) sections 1 and 3; the `agent_data_duckdb` function-registration patterns (`agent_data_duckdb/src/lib.rs`, `agent_data_duckdb/src/vtab.rs`); [flockmtl](https://duckdb.org/community_extensions/extensions/flockmtl) as a precedent for resource-management SQL surfaces (model/prompt declarations) in an extension.
- **Dependencies.** Realises the model from component 1; constrains the storage schema (component 3, which must serve these access paths) and fixes the call sites where component 4's embedding/LLM boundary is invoked.

### 3. Storage, indexing, and retrieval implementation

- **Scope.** The physical realisation inside the DuckDB database: the native table schema per tier (including `ARRAY(FLOAT, N)` embedding columns and how the dimension is configured); scope keys as ordinary indexed columns so scope filtering is a pre-filtering `WHERE` clause pushed into every retrieval — the correctness-critical property [DUCK-R2](../research/DUCK-R2-reference-architectures.md) carries over from KAOS; the hybrid retrieval implementation (exact `array_cosine_distance` scans as the always-correct default, the opt-in `vss` HNSW acceleration path and its experimental-persistence caveat, `fts` BM25 and its reindex-on-change constraint, and the fusion method such as reciprocal-rank fusion or weighted scoring); index lifecycle policy (when the FTS index is rebuilt, when HNSW compaction runs); and schema versioning/migration for the extension's own tables.
- **Key decisions.** The settled direction — reuse `vss` + `fts` for the initial version — recorded with its consequences: at what data volume brute-force stops sufficing and HNSW becomes worth its persistence caveat, and whether HNSW is enabled by default for in-memory databases but opt-in for file-backed ones; the FTS rebuild policy (on-write, staleness-threshold, or explicit) given the index does not auto-update; the fusion formula and how the scoring weights from component 1 map onto it; and how the schema migrates across extension versions (a version table and idempotent migrations on load).
- **Primary inputs.** [DUCK-R1](../research/DUCK-R1-duckdb-extension-ecosystem.md) sections 1 and 2 (the `vss`/`fts` capabilities and caveats); [DUCK-R2](../research/DUCK-R2-reference-architectures.md) section 3; [KAOS-R11](../../memory/research/KAOS-R11-short-term-memory-storage.md) for relational short-term storage findings.
- **Dependencies.** Implements components 1 and 2 beneath the SQL surface; provides the vector columns component 4 fills; its `vss`/`fts` dependency choice feeds the packaging decisions in component 5.

### 4. Embedding and LLM integration boundary

This component is the deliberately open decision: how much of the intelligence loop lives inside the engine versus with the host. It must be an explicit trade-off record with pros, cons, and a recommendation, because it determines whether DuckMemory is a self-contained end-to-end memory engine or a storage/retrieval core with a bring-your-own-models contract.

- **Scope.** Where embedding generation happens (host passes vectors in; the extension calls an OpenAI-compatible HTTP endpoint; or the extension runs local inference, quackformers-style); where LLM-driven work happens (fact extraction from folded turns and digest summarization — host-side, or in-engine flockmtl-style so a single SQL call performs write → extract → consolidate); how models and credentials are configured if calls are in-engine (DuckDB secrets manager, extension settings, per-store model declarations); and the degradation contract when a model endpoint is unavailable (writes must still land; extraction defers).
- **Key decisions.** The core trade-off: **pure storage/retrieval** maximises portability, determinism, and testability but pushes the entire pipeline onto every host; **embeddings in-engine** makes recall and write self-contained (text in, memories out) at the cost of network calls inside query execution; **full end-to-end in-engine** delivers the strongest "memory engine in a database" story — matching the project's end-to-end goal — but embeds LLM latency, cost, and failure modes inside SQL statements and complicates the concurrency picture from component 2. A layered contract (host-provided vectors always accepted; in-engine calls as an optional configured convenience) is a candidate recommendation the ADR must weigh against the added surface. Also: whether extraction quality justifies in-engine prompting at all, or extraction stays a host recipe with the engine owning only storage-side consolidation (digest folding without an LLM, e.g. windowed verbatim eviction).
- **Primary inputs.** [DUCK-R1](../research/DUCK-R1-duckdb-extension-ecosystem.md) section 3 (flockmtl, quackformers, llm precedents proving each option is technically viable); [memory adr_0002](../../memory/adrs/adr_0002_memory-implementation-mem0-and-pydantic-ai-integration.md) for the extraction-pipeline shape being relocated; [DUCK-R2](../research/DUCK-R2-reference-architectures.md) section 1 on the cascade write model.
- **Dependencies.** Plugs into the call sites fixed by component 2 and fills the vector columns defined by component 3; its outcome materially changes the scope of component 5 (an HTTP-calling or inference-running extension carries heavier dependencies and a larger test matrix).

### 5. Extension architecture, packaging, and distribution

- **Scope.** The engineering shell around the engine: the Rust crate architecture in the new repository (the `agent_data_duckdb` skeleton — `cdylib` on `duckdb`/`libduckdb-sys` with the `loadable-extension` feature, `#[duckdb_entrypoint_c_api()]` entrypoint, a generic function-registration layer, `extension-ci-tools` Makefile build, `duckdb-release.toml` version pinning); how the `vss` and `fts` dependencies are declared and loaded (autoload behaviour, failure mode when unavailable); the testing strategy (SQLLogicTest suites with deterministic fixtures, plus Rust unit tests for scoring/fusion logic, plus — depending on component 4 — mock model endpoints); CI and the platform build matrix; and distribution through the DuckDB community-extensions repository (descriptor PR, central signed builds, `INSTALL duckmemory FROM community`).
- **Key decisions.** The settled direction — Rust, new dedicated repository — recorded with its consequences, chiefly the C-API boundary from [DUCK-R1](../research/DUCK-R1-duckdb-extension-ecosystem.md) section 4 (no custom index types, hence the component 3 reuse decision); repository naming and layout; how the extension's DuckDB version pinning tracks upstream releases; the SQLLogicTest fixture strategy for a *stateful* extension (unlike `agent_data_duckdb`'s read-only functions, tests must create, mutate, and re-open memory stores); release versioning and its coupling to the schema migrations from component 3; and whether community distribution is targeted from the first release or after the MVP hardens.
- **Primary inputs.** [DUCK-R2](../research/DUCK-R2-reference-architectures.md) section 2 (the full skeleton inventory); [DUCK-R1](../research/DUCK-R1-duckdb-extension-ecosystem.md) sections 4 and 5.
- **Dependencies.** Depends on all preceding components for what must be built and tested; constrains none of them conceptually, but its C-API boundary is the standing feasibility check every other component's decisions must pass.

## Cross-cutting concerns

These are not separate components; each ADR addresses them within its own scope rather than deferring them to a single record.

- **Observability.** An embedded engine has no metrics endpoint; introspection is SQL. Each ADR states what its surface exposes for inspection (stats table functions, access counters on the envelope, consolidation history), so a host can observe memory behaviour with plain queries.
- **Concurrency and process model.** DuckDB permits a single writer process; agent hosts may hold multiple connections or reopen files across sessions. Component 2 fixes the transactional semantics, but every component states how its state behaves under reopen, checkpoint, and concurrent readers — in particular anything touching the `vss` experimental-persistence caveat.
- **Security posture.** There is no trusted server boundary: scope is a correctness and data-organization feature, not an authentication mechanism against the embedding process. Credentials for any in-engine model calls (component 4) go through DuckDB's secrets manager, never into extension tables.
- **Consumer interoperability.** KAOS interop is a non-goal for the initial version, but component 1's model should not preclude a host framework mapping its own scope/tier contract onto DuckMemory; each ADR notes where a decision would close that door.

## Dependency and sequencing

The components form a dependency chain rather than a flat list, but they are authored in the order that lets each decision rest on a settled foundation. Component 1 (the memory model) defines what memory means and the structure of scope, tiers, and lifecycle; component 2 (the SQL interface and execution model) fixes how a host invokes it and where daemonless consolidation runs; component 3 (storage, indexing, retrieval) realises both inside DuckDB's native tables and the `vss`/`fts` engines; component 4 (the embedding/LLM boundary) is decided once the surfaces it connects are stable, because its options are meaningless without fixed call sites and vector columns; and component 5 (architecture, packaging, distribution) wraps the result in a buildable, testable, distributable extension. The recommended authoring order is therefore 1, 2, 3, 4, 5 — with the cross-cutting concerns settled inside each.

## ADR conventions

- **Filename.** ADRs use `adr_NNNN_<slug>.md` with a zero-padded sequence number and a short kebab-case slug (for example `adr_0001_memory-model-and-lifecycle-operations.md`). This index uses the descriptive name `adr_high_level_components.md` and is not itself a numbered decision.
- **Numbering.** Numbers are assigned in authoring order and are provisional until an ADR is written; additional ADRs may slot in as decisions are split out. The numbering reflects the order decisions are recorded, not a strict mapping to the five components.
- **Status lifecycle.** Each ADR carries a status: `Proposed`, `Accepted`, `Superseded` (by a named later ADR), or `Deprecated`. Superseding never edits the original record; it adds a new one and updates the status.
- **Structure.** Each ADR follows a consistent shape: title and status; context (the problem and the relevant research, cited); the decision; consequences (positive, negative, and follow-on work); and alternatives considered with the reason each was not chosen. References link back to the research documents rather than restating them.
- **Authoring style.** ADRs follow the repository [CONVENTIONS](../CONVENTIONS.md): single-line paragraphs and list items, Markdown links for cross-references, and no references to phases, tasks, or plan steps in the document body.
