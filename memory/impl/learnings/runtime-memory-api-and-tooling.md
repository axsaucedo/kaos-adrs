# Runtime memory API and tooling — finalized learnings

## Purpose

This document records the runtime memory API, tooling, topology, ownership, and packaging decisions reached during review of the KAOS runtime memory client. It is intended as a durable design record for the runtime, the memory service boundary, and the operator wiring that will make those runtime choices safe in Kubernetes, while preserving the existing learning-doc style of explicit decisions, rationale, rejected alternatives, and consequences.

## Decision 1: the runtime HTTP backend is `RemoteMemory`, not `ServiceMemory` or `Mem0Client`

The runtime HTTP memory backend is named `RemoteMemory`. The name describes the role precisely: it is the runtime-side client for the remote KAOS-owned central memory service, and it speaks the KAOS tiered `/v1/recall`, `/v1/write`, and `/v1/forget` contract with KAOS scope payloads, context blocks, and failure-mode semantics.

The rejected name `ServiceMemory` was directionally correct but too vague. It described that another service is involved, but it did not say what kind of runtime memory backend this object is relative to the other implementations. The accepted name sits cleanly beside the `Memory` ABC siblings: `LocalMemory` for in-process short-term replay, `NullMemory` for disabled memory, `RedisMemory` for the legacy distributed short-term backend, and `RemoteMemory` for the remote tiered service client.

The rejected name `Mem0Client` would be actively wrong. Mem0 has two distinct client surfaces: `mem0.Memory`, the open-source in-process engine that KAOS embeds inside the central memory service as the long-term fact tier, and `mem0.MemoryClient`, the HTTP client for Mem0's hosted platform. KAOS does not use Mem0's hosted platform client, does not scope any Mem0 SaaS work here, and does not expose Mem0's hosted schema at the runtime boundary.

The consequence is a clear layering rule. Runtime code talks to `RemoteMemory`; the memory service may use `mem0.Memory` internally for long-term extraction and retrieval; there is no Mem0 platform client in KAOS; and the runtime API name must not imply that an agent pod is calling Mem0 directly.

## Decision 2: enabling memory always applies the automatic baseline, while `memory.tools` adds optional agent-driven tools

The previous `recall.presentation` model combined two separate concerns into one knob with `block`, `tools`, and `both` values. That shape made the recalled block feel like one presentation option among others and allowed a tool-only mode where memory could be enabled without automatic recall injection. The new model separates baseline behaviour from optional agent agency.

When memory is enabled, KAOS always applies the automatic baseline: before each run, the runtime recalls memory and injects the returned context block whenever the service returns one; after the run, the runtime flushes the run's new turns for extraction and short-term continuity. The recalled block is no longer gated by a presentation mode. It is the default consequence of enabling memory because KAOS-on-Mem0 is meant to follow the idiomatic automatic search-before-LLM and add-after-turn pattern rather than relying on the model to remember to call a memory tool.

On top of that baseline, `memory.tools` is additive. `memory.tools: all` exposes both `save_memory` and `search_memory`; `memory.tools: read` exposes only `search_memory`; `memory.tools: write` exposes only `save_memory`; and an unset value exposes no explicit memory tools. The axes are symmetric and readable: read means on-demand retrieval through `search_memory`, and write means on-demand save through `save_memory`.

The rejected alternative was to preserve a tool-only, no-automatic mode similar to pure Letta/MemGPT operation. KAOS deliberately drops that mode for now because it is not the natural default for a Mem0-backed system: Mem0's strongest ergonomic pattern is automatic app-layer recall and extraction, and making “enabled” mean “automatic” gives users useful memory without requiring prompt-level discipline from every agent. If pure agent-driven memory becomes necessary later, it should return as an explicit escape hatch such as `automatic: false`, not as an overloaded presentation value.

The consequence is simpler operator and runtime policy. The runtime can inject a block whenever one is present, the operator can validate tools independently of the baseline, and users can reason about explicit tools as additional powers rather than a replacement for the safety net.

## Decision 3: the framework survey supports a two-axis model for automatic and tool-driven memory

The saved framework survey shows that memory design is best understood as two independent axes: write can be automatic extraction or an explicit agent tool, and read can be automatic injection or an on-demand search tool. The KAOS design follows that model by making automatic read and write the baseline while allowing optional explicit read and write tools.

For Mem0, the survey found that the idiomatic write path is automatic extraction. The `Memory.add(messages, infer=True)` path runs Mem0's own extraction pipeline: it gathers recent session context, retrieves related vector memories, asks its extraction LLM what new facts to add, then deduplicates, embeds, links entities, and stores the result. The survey also recorded that `infer=False` stores raw transcripts and that Mem0 documentation warns this is for raw transcript workflows rather than the usual structured-memory path. Official LangChain and LangGraph examples call `mem0.add(...)` at the application layer after each interaction, not as a model-selected tool.

For Mem0 reads, the survey found the matching automatic pattern: application code calls `search()` before the LLM call and injects the returned memories into the prompt. The report cites Mem0's own “how it works” guidance that most applications call `add` after a useful interaction and call `search` before a model call, plus LangGraph and LangChain examples that retrieve context before invoking the model. The same survey also records that Mem0 offers optional explicit tools through LangChain `StructuredTool` integrations such as `add_memory` and `search_memory`, and through Mem0 MCP tools such as `add_memory` and `search_memories`; these are valid alternatives, but they are not the baseline SDK pattern.

For LangMem, the survey found a deliberately configurable dual model. The hot-path pattern exposes `create_manage_memory_tool` and `create_search_memory_tool`, letting the LLM decide when to write and search. The background pattern uses `create_memory_store_manager` or `ReflectionExecutor` so application code runs extraction outside the main response path. The survey cites LangMem's documented latency and quality tradeoff table: active hot-path memory has higher latency, immediate updates, and processing during the response, while background memory has no response latency, delayed updates, and between-call processing. This is the clearest cross-framework articulation of why KAOS should not make explicit tools the only way to form memory.

For Letta and MemGPT, the survey found the pure explicit-tool archetype. Core memory blocks are always in context and are edited by tools such as `core_memory_append`, `core_memory_replace`, and related block-edit operations, while archival memory is written and searched with explicit operations such as `archival_memory_insert` and `archival_memory_search`. The agent is the actor deciding when and what to remember, and the large archival tier is searched on demand rather than automatically injected on every turn.

For LlamaIndex, the survey found memory to be infrastructure-level rather than tool-level in the default pattern. The newer `Memory` class uses FIFO pressure and memory blocks; application code puts messages into memory, memory blocks process overflow, and `aget()` content is injected automatically as system or user context. The survey did not find a default explicit write or search tool exposed to the LLM.

For Pydantic AI, the survey found no first-party memory abstraction. Pydantic AI provides message-history plumbing through `Agent.run(message_history=...)` and tool registration through `@agent.tool` or `tools=`, but long-term memory, retrieval, extraction, and memory tools are developer-implemented. This matters because KAOS owns the integration pattern rather than conforming to a hidden Pydantic AI memory system.

KAOS therefore aligns with Mem0's dual model rather than Letta's pure explicit-tool model: automatic recall injection and automatic post-run write form the reliable default, while tools remain available for cases where agent agency adds value. `save_memory` remains useful when the user explicitly asks the agent to remember a preference and the agent should acknowledge the save, such as “remember I prefer metric units”; when the interaction is one-sided or unusual enough that an end-of-run automatic flush may lack the assistant-side context normally used for extraction; and when the agent judges an episodic “note this” event to be salient enough to save directly. `search_memory` remains useful when the user explicitly asks what the agent remembers about a topic, when a targeted multi-hop query needs more precise retrieval than the automatic pre-run query, and when transparency or debugging requires showing the retrieved memory surface rather than silently injecting it.

The consequence is that `memory.tools` is not an admission that automatic memory is insufficient. It is a narrow agency layer over a reliable automatic baseline, matching the strongest Mem0 pattern while preserving the useful parts of explicit-tool systems.

## Decision 4: supported runtime topology collapses to `LocalMemory`, `RemoteMemory`, and `NullMemory`

The supported runtime memory topology is intentionally small. `enabled: false` maps to `NullMemory`. Enabled memory without a `memoryStore` reference maps to `LocalMemory`, which provides in-process short-term recent-turn replay only, with no long-term tier, no cross-pod sharing, and loss on restart. Enabled memory with a `memoryStore` reference maps to `RemoteMemory`, which talks to the central memory service and provides the full short-term, medium-term, and long-term tier contract.

The switch is the presence of the `memoryStore` reference on the Agent memory block. If `memoryStore` is set, the operator injects the service endpoint and the runtime constructs `RemoteMemory`; if `memoryStore` is absent, the runtime uses `LocalMemory`. There is no separate backend enum because a second selector would create invalid combinations, duplicate the meaning of the reference, and force the operator to arbitrate conflicts that the resource shape can avoid.

Two alternatives were rejected as over-engineering. An embedded backend that places the Mem0 engine inside every agent process would multiply extraction executors, model calls, vector-store configuration, migrations, and failure surfaces across pods; it would also blur the service ownership boundary that keeps long-term memory centralized. A per-pod memory sidecar would preserve process separation but still duplicate service instances per agent pod, complicate lifecycle and scope ownership, and offer little over a central service for a system whose memory semantics are meant to cross restarts and replicas.

The consequence is that validation must fail closed rather than degrade silently. A user-scope or shared-scope agent requires a `memoryStore` because `LocalMemory` is pod-local and cannot provide cross-agent or cross-replica semantics; if multiple replicas each hold a private local window, they are divergent copies rather than one scope. Setting `memory.tools` also requires a `memoryStore` because `save_memory` and `search_memory` target the long-term tier, which `LocalMemory` does not have; exposing those tools without a store would produce silent no-ops or misleading results. The operator should reject both misconfigurations so that scope ownership and tool semantics remain honest.

## Decision 5: private scope owner derivation fails closed unless it resolves to a concrete qualified agent identity

Private-scope memory must resolve to a concrete, agent-unique owner. Returning an ambiguous owner when the runtime lacks identity is unsafe because every identity-less agent would collapse onto the same empty or default owner and silently share one private partition. That failure mode is a cross-agent memory leak disguised as a fallback.

The runtime scope derivation therefore raises when it cannot derive an unambiguous private owner. The correct owner is the fully qualified AIB identity `kaos://agent/{namespace}/{name}`, not a name-only identifier. Name-only identifiers collide across namespaces, while the qualified identity is already the security identity the operator mints as `AGENT_AUTH_IDENTITY`.

The rejected alternative was to keep a permissive local fallback, for example deriving private ownership from only the agent name or returning a placeholder when identity is missing. That would make early tests convenient but would encode unsafe semantics into production runtime code. A private partition that is sometimes shared is worse than no memory because it can leak context between agents while appearing to be correctly isolated.

The split of responsibility is explicit. The runtime memory client owns the fail-closed guard when private scope ownership is ambiguous. The operator wiring must guarantee the qualified identity is always injected and must drop any name-only local fallback; that wiring belongs with the M4 memory-store control-plane plan because it is the CRD-to-runtime identity projection, not because the runtime should ever accept ambiguous private ownership.

## Decision 6: `kaos-memory` should own the client, shared contract models, and optional Pydantic AI integration

The current package boundary is asymmetric. The `kaos_memory` package contains the memory service, while the consuming runtime hand-rolls `RemoteMemory` and duplicates the wire contract for scope payloads, recall requests and responses, write requests and responses, and forget requests and responses. That duplication lets the server and client drift even though they are two sides of the same KAOS-owned API.

The direction is for the `kaos-memory` library to own the service, the shared contract models, and the client as one source of truth. In that shape, `RemoteMemory` in the runtime becomes a thin adapter from the runtime's `Memory` ABC to the shared client, rather than a second implementation of the HTTP schema.

The library should also ship a Pydantic AI integration as an optional extra, `kaos-memory[pydantic-ai]`, while keeping the core package framework-agnostic. The integration module owns the message adapters `messages_to_turns` and `turns_to_message_history`, which are the bidirectional bridge currently represented in runtime code as `pydantic_message_to_turns` and `reconstruct_message_history`. It should also own the `save_memory` and `search_memory` toolset so a Pydantic AI consumer imports one coherent feature set instead of assembling protocol models, adapters, and tools from different packages.

The only meaningful design concern is that this integration makes the library import `pydantic_ai` types. The accepted answer is packaging, not a redesign: the core library remains free of Pydantic AI imports, and the integration lives behind the optional extra. The functions themselves are pure and stateless, so the carve-out is not deferred because of algorithmic difficulty; it is sequenced as a dedicated packaging change so it does not destabilize the in-review runtime client work, and it pairs naturally with moving the client into the shared library.

The consequence is a cleaner ownership boundary. The memory service and client evolve together, the runtime stops redefining the wire contract, Pydantic AI consumers get adapters and tools from the same package that defines the memory API, and future contract changes become type-level library changes rather than coordinated hand edits across repos.

## Open item: recall block content may duplicate reconstructed message history

The remaining open question is whether the injected recall block should include medium-term digest and recent-turn content when the runtime also reconstructs `message_history` from the same short-term and medium-term tiers. In the current block presentation, conversational continuity can appear twice: once inside the injected memory block and again as reconstructed history.

The leading option is to make the service block carry long-term facts only, while conversational continuity comes solely from reconstructed message history. That would preserve the block as durable memory context, keep the model's recent conversation in the native Pydantic AI history channel, and reduce duplicate token spend. The tradeoff is that it touches both sides of the boundary: the service block assembly would need to stop serializing continuity content into the block, and the runtime injection path would need to rely on reconstructed history for that continuity.

This is intentionally parked until the `kaos-memory` client and Pydantic AI integration carve-out is sequenced. The duplication question crosses the service contract, the shared message adapters, and the runtime injection path, so resolving it after the contract boundary is consolidated will reduce churn and make the eventual behaviour easier to test.
