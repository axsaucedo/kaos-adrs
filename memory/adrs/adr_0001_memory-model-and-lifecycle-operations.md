# ADR 0001 — Memory model and lifecycle operations

- **Status.** Proposed
- **Date.** 2026-06-29
- **Component.** 1 of 5 ([adr_high_level_components](./adr_high_level_components.md))
- **Depends on.** [KAOS-R1](../research/KAOS-R1-memory-features-and-limitations.md), [KAOS-R2](../research/KAOS-R2-memory-ecosystem-research.md), [KAOS-R7](../research/KAOS-R7-target-picture.md), [KAOS-R8](../research/KAOS-R8-production-memory-adoption-in-frameworks.md), [KAOS-R9](../research/KAOS-R9-procedural-temporal-memory-in-agent-systems.md)
- **Resolved by.** [ADR 0002](./adr_0002_memory-implementation-mem0-and-pydantic-ai-integration.md) (engine = Mem0, tier coverage), [ADR 0003](./adr_0003_memory-interface-and-runtime-data-plane.md) (runtime data plane, short-term tier bound), [ADR 0004](./adr_0004_deployment-topology-and-control-plane.md) (topology, execution model, control plane), and [ADR 0005](./adr_0005_multi-tenancy-agent-grouping-and-governance.md) (scope grammar, A2A binding, forgetting). The placeholders this record originally carried are now closed by those records and folded into the decisions below.

## Context

KAOS today exposes a single short-term-memory tier through the `Memory` abstraction (Local, Redis, Null backends), bounded to the live per-run message history bridged into Pydantic AI. There is no long-term tier, no scope dimension beyond a session/user key, and no consolidation, retrieval scoring, or forgetting. The target picture ([KAOS-R7](../research/KAOS-R7-target-picture.md)) calls for an engine-agnostic memory model with multiple conceptual lanes, hierarchical scoping, and lifecycle operations, with the implementing engine chosen separately so the model stays durable while engines remain swappable.

This ADR settles only the conceptual memory model and the lifecycle operations over it: the taxonomy and which tiers are committed, the metadata envelope, the scope-addressing structure including A2A delegation propagation, and the semantics of retrieval, consolidation, and forgetting. It does not pick an engine, a topology, or a CRD; those are later records. Several parameters here were originally left as engine-dependent policy; with [ADR 0002](./adr_0002_memory-implementation-mem0-and-pydantic-ai-integration.md) through [ADR 0005](./adr_0005_multi-tenancy-agent-grouping-and-governance.md) now settled, the decisions below fold in their concrete realisation on the Mem0 engine while keeping the conceptual model engine-agnostic and durable.

## Decision

### Memory tiers (capability-tiered, not all-at-once)

Short-term memory is committed unconditionally: it is table stakes, owned by KAOS and Pydantic AI as the live per-run message-history bridge, and is never delegated to an external engine regardless of later choices. Semantic and episodic memory together form the committed long-term core, matching the production norm of collapsing them into one store while keeping them distinguishable in metadata ([KAOS-R8](../research/KAOS-R8-production-memory-adoption-in-frameworks.md), [KAOS-R9](../research/KAOS-R9-procedural-temporal-memory-in-agent-systems.md)). [ADR 0002](./adr_0002_memory-implementation-mem0-and-pydantic-ai-integration.md) fixes the engine as Mem0, which realises exactly this collapsed semantic-plus-episodic long-term store over a single vector backend with LLM-driven fact extraction, so the committed tier set is **short-term plus a unified long-term store** and nothing more in this version. Temporal (bi-temporal validity) and procedural (skills) are deferred as later capability tiers: temporal is delivered rigorously only by Graphiti-class engines and Mem0 carries only coarse `created_at`/`updated_at` timestamps rather than true bi-temporal validity, while procedural is coding-agent-centric and the least standardised. Even without a dedicated temporal query API, recall remains available through vector and text search so questions of the form "who was president in 2002" resolve from the long-term store's content.

### Scope-addressing structure (hybrid hierarchical path)

Scope is a hierarchical, segmented path with prefix-sharing: a small set of canonical roots (tenant, user, agent, session) plus a deployment-defined segmented path, where two scopes sharing a prefix share memory above the divergence point and are isolated below it. The path anchors to stable, verifiable identities rather than ephemeral keys — the user principal and the agent's AIB `client_id` — so memory persists across redeploys and stays auditable; a per-deployment UUID is insufficient because it orphans memory on restart and is not verifiable across agents. [ADR 0005](./adr_0005_multi-tenancy-agent-grouping-and-governance.md) realises this concretely against the Mem0 engine as a **flat owner-key model** rather than a composed prefix string: the operator-facing surface is a four-value `scope` enum (`private`, `user`, `shared`, `session`) on the Agent memory block, and the memory service maps each value onto exactly one of Mem0's native owner keys — `private` to `agent_id` (the agent's fully-qualified identity), `user` to `user_id` (the principal), `session` to `run_id` (the live run), and `shared` to `agent_id` set to a reserved store-wide constant (`kaos:shared`) that is deliberately distinct from any real agent identity so a shared write can never collide with a private one. `private` and `shared` therefore share the `agent_id` key and differ only by value — unique-per-agent versus one fixed sentinel. An operation whose required owner key is unresolved fails closed rather than widening to an unscoped query. The MemoryStore boundary is itself the top-level group, so the canonical roots collapse to "store / owner / session" in practice without a separate grouping resource.

### A2A delegation propagation

Today an agent delegating to another passes its full session short-term memory and nothing else. The model extends this so the delegate inherits a configurable scope prefix by default — shared above, isolated below — making cross-agent sharing follow the same prefix semantics as static scopes. [ADR 0005](./adr_0005_multi-tenancy-agent-grouping-and-governance.md) binds this to the existing delegation surface: when an agent delegates, the server injects the delegator's scope prefix into the delegate's request context through the `DelegationToolset` path (`pydantic-ai-server/pais/tools.py`), so the delegate reads and writes under the inherited prefix without trusting client-supplied scope. The default is shared-above/isolated-below; the binding is overridable per delegation edge to full-share (identical scope) or full-isolation (delegate gets only its own private scope).

### Metadata envelope (universal core, per-tier extensions)

Every item carries a universal core envelope — id, scope path, event and ingestion time, provenance, importance, access stats, retention/pin policy, trace linkage, identity anchors (principal, agent `client_id`) — plus optional per-tier extension fields. No multimodality at this stage; the envelope assumes text. Conflict resolution, importance scoring, and aliasing are engine-dependent and deferred to ADR 0002.

### Retrieval, consolidation, forgetting

Retrieval is configurable weighted relevance with importance and recency decay, with a reranker option, defaulted to fixed sensible values; consolidation is hybrid threshold/event-driven plus periodic, always background; forgetting offers decay, importance threshold, pinning, non-lossy temporal invalidation, and hard erasure as declarative policy. [ADR 0002](./adr_0002_memory-implementation-mem0-and-pydantic-ai-integration.md) and [ADR 0003](./adr_0003_memory-interface-and-runtime-data-plane.md) fix how these are realised on Mem0: recall is Mem0's relevance search over the long-term store (vector similarity, optional reranking) returned to the agent as recalled context; consolidation is Mem0's own LLM-driven fact extraction and ADD/UPDATE/DELETE reconciliation, run as background work after each turn ([ADR 0004](./adr_0004_deployment-topology-and-control-plane.md) fixes this as in-process fire-and-forget extraction); and forgetting is Mem0 deletion plus the scope-targeted synchronous erasure fan-out across both tiers defined in [ADR 0005](./adr_0005_multi-tenancy-agent-grouping-and-governance.md). Importance scoring, pinning, and bi-temporal invalidation beyond Mem0's coarse timestamps remain forward-looking policy not exercised in this version.

### Short-term tier bound

The short-term tier is bounded by a configurable **token budget** rather than turn-count eviction, paired with a configurable **rolling summary** that compacts older turns once the budget is exceeded so salient context survives truncation. [ADR 0003](./adr_0003_memory-interface-and-runtime-data-plane.md) fixes both as first-class, per-agent-configurable behaviours, and [ADR 0004](./adr_0004_deployment-topology-and-control-plane.md) places their execution in the memory service (the same summarization model role drives both Mem0 extraction and the rolling summary). The raw turn is always persisted before summarization, so a lost or deferred summary is re-derivable.

## Consequences

Positive: a durable, engine-agnostic model that commits the high-certainty tiers and, with the engine and topology decisions now landed, pins each lifecycle operation to a concrete Mem0 realisation while keeping the conceptual surface swappable; identity anchored to AIB makes scopes verifiable and survivable. Negative: temporal and procedural tiers stay deferred, so use cases needing rigorous bi-temporal validity or skill memory are not served in this version and will require a future engine extension. Follow-on: [ADR 0002](./adr_0002_memory-implementation-mem0-and-pydantic-ai-integration.md) fixes tiers, retrieval, consolidation, forgetting, and conflict resolution; [ADR 0005](./adr_0005_multi-tenancy-agent-grouping-and-governance.md) finalises grouping and the A2A binding.

## Alternatives considered

Commit all five tiers immediately — rejected: temporal and procedural depend on engine capabilities not yet chosen. Discrete fixed scope levels only — rejected for the hybrid path's flexibility. Random UUID scope keys — rejected: not stable or verifiable across redeploys and A2A.
