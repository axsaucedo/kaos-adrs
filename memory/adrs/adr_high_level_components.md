# KAOS memory ADRs — high-level components

This document opens the architecture-decision phase for KAOS memory. The research stages ([KAOS-R1](../research/KAOS-R1-memory-features-and-limitations.md) through [KAOS-R7](../research/KAOS-R7-target-picture.md)) established the current implementation and its gaps, the ecosystem, the tooling landscape, a selection, and an unrestricted target picture. The target picture describes the components and conceptual memory lanes at a high level; this document decomposes that picture into the **high-level components that each require an architecture decision record (ADR)**, so the design work can proceed component by component.

The intent here is deliberately *coarse*. The components below are bundled rather than finely segregated: each one groups a set of closely-coupled decisions that are best made together, so the ADR set stays small and coherent instead of fragmenting into many narrow records. Each component listed will become one ADR (occasionally a small number, if a decision later proves too large to bundle), and this document is the index that anchors them.

## How this relates to the research

The components are derived directly from the target picture in [KAOS-R7](../research/KAOS-R7-target-picture.md), grounded in the requirements baseline [KAOS-R1](../research/KAOS-R1-memory-features-and-limitations.md) and the ecosystem vocabulary [KAOS-R2](../research/KAOS-R2-memory-ecosystem-research.md), and constrained by the selection in [KAOS-R6](../research/KAOS-R6-tool-selection.md). Where an ADR weighs an option, the per-tool deep dives ([KAOS-R5-1](../research/KAOS-R5-1-mem0.md) through [KAOS-R5-5](../research/KAOS-R5-5-redis-agent-memory-server.md)) are the evidence base. ADRs decide; they do not re-litigate the research. Each ADR records what was chosen, why, and the consequences, citing the research rather than repeating it.

The research remit was *informed but not constrained* by the available tooling. The ADRs are the opposite: they are where the unrestricted target is reconciled with what is buildable and adoptable now, producing concrete, committed decisions with a status lifecycle.

## Components requiring an ADR

The five components below cover the full target surface. They are listed in authoring order rather than strictly by dependency: the conceptual memory model and the engine that implements it are settled first, the runtime interface that realises them follows and dives deeper into implementation, the topology and control plane that operate them come next, and the cross-cutting tenancy, grouping, and governance model closes the set. Cross-cutting concerns that belong to no single component are listed separately afterwards.

### 1. Memory model and lifecycle operations

- **Scope.** The conceptual memory lanes and the operations over them: the target taxonomy (working, episodic, semantic, profile, relational/temporal, procedural) and which tiers are committed and in what order; the common metadata envelope every item carries (id, scope, event/ingestion time, provenance, importance, access stats, retention/pin policy, trace linkage) and the structure of the scope dimension itself, including how scopes are derived across agent delegation; the retrieval scoring model (hybrid recall scored by relevance, importance, and recency with decay); consolidation/reflection; and forgetting/retention policy semantics.
- **Key decisions.** Which tiers are in scope versus deferred, and their sequence; the metadata-envelope schema that keeps tiers interoperable and the engine swappable; the structure of the scope dimension — whether scope is a fixed set of discrete levels (session, user, agent, tenant) or a hierarchical, segmented scope path that lets a deployment define its own levels and choose where sharing happens, for example a path of the form `tier-X:tier-Y:agent` where one agent addressed `X:Y:A` and another addressed `X:Z:B` share memory at the `X` level but remain isolated at `Y` versus `Z` and at the agent level below (sharing at a common prefix, isolation past the point the paths diverge), and how a scope is derived when one agent delegates to another — whether a delegated agent inherits the delegator's scope prefix, is mapped onto a shared group level, or is given a fresh isolated path, so cross-agent and A2A sharing follow the same prefix semantics; the retrieval scoring formula and its configurability; the consolidation trigger and schedule model; and the forgetting primitives (decay, pinning, non-lossy temporal invalidation, hard erasure) and how they are expressed as policy.
- **Primary inputs.** [KAOS-R7](../research/KAOS-R7-target-picture.md) memory model and retrieval/consolidation/forgetting; [KAOS-R2](../research/KAOS-R2-memory-ecosystem-research.md) taxonomy and scoping model; [KAOS-R5-2](../research/KAOS-R5-2-zep-graphiti.md) temporal model; [KAOS-R5-4](../research/KAOS-R5-4-memobase.md) profiles; [KAOS-R8](../research/KAOS-R8-production-memory-adoption-in-frameworks.md) which tiers are actually used in production; [KAOS-R9](../research/KAOS-R9-procedural-temporal-memory-in-agent-systems.md) precise tier definitions and the semantic-versus-temporal boundary.
- **Dependencies.** The conceptual foundation. Defines what the runtime interface (component 3) must carry and what the engine (component 2) must support; its policy fields surface on the control-plane CRD (component 4); and the scope-addressing model it defines — including its delegation-propagation rules — is consumed by the runtime interface (component 3, scope injection) and by agent grouping (component 5).

### 2. Memory implementation: engine selection and tier coverage

- **Scope.** The decision of how memory is actually implemented across both the short-term (working) and long-term tiers: the build-versus-adopt boundary; which engine or engines are adopted and how much of the memory contract each one backs — ideally serving as the primary implementation for as many tiers as it can, up to and including working memory, rather than being confined to long-term; the adapter contract that maps the memory interface onto a concrete engine; how a backend is swapped; where embedding and extraction models bind (references to KAOS `ModelAPI` resources); and where KAOS must supplement engine gaps (OpenTelemetry instrumentation, scope and tenancy enforcement).
- **Key design choices.** Per-engine tier coverage and the synergies and trade-offs it creates: a two-tier engine such as the Redis Agent Memory Server ([KAOS-R5-5](../research/KAOS-R5-5-redis-agent-memory-server.md)) can realistically back both working and long-term memory as a single implementation, whereas a long-term/fact-centric engine such as Mem0 ([KAOS-R5-1](../research/KAOS-R5-1-mem0.md)) has no live conversation-buffer concept and must be paired with a KAOS-owned working-memory backend. The decision therefore covers whether one engine spans both tiers or the tiers are segregated across implementations; the one firm invariant that KAOS and Pydantic AI always own the live per-run message-history bridge regardless of engine; confirming the primary engine and how multiple engines coexist behind one interface; how model references flow into the engine; and the explicit list of capabilities KAOS wraps around the engine rather than inheriting from it.
- **Primary inputs.** [KAOS-R6](../research/KAOS-R6-tool-selection.md) selection; [KAOS-R5-1](../research/KAOS-R5-1-mem0.md) Mem0; [KAOS-R5-5](../research/KAOS-R5-5-redis-agent-memory-server.md) Redis AMS two-tier model; [KAOS-R7](../research/KAOS-R7-target-picture.md) design principles and the capabilities-no-tool-provides section; [KAOS-R8](../research/KAOS-R8-production-memory-adoption-in-frameworks.md) the production norm of an external engine backing memory.
- **Dependencies.** Depends on the operations and tiers defined in component 1; constrains the runtime interface (component 3) and the topology and control plane (component 4). Closely coupled to component 1 — the model says what must be stored and recalled, the implementation says what an engine can back and how the tiers are split or unified — but kept as a separate record because the build-versus-adopt and adapter-contract decisions stand on their own.

### 3. Memory interface and runtime data plane

- **Scope.** The `Memory` abstraction in the Pydantic AI runtime and its extension from today's working-memory-only contract into a tiered interface (working memory plus long-term operations — write/extract, search/recall, consolidate, forget), where each tier may be backed by a KAOS-owned backend or by the adopted engine depending on the tier coverage decided in component 2. Includes the recall and write hooks around an agent run, the strategy for injecting recalled context (a structured memory block versus tools), the Pydantic AI message-history bridge and its fidelity fixes (replaying tool calls, summarizing rather than truncating), the point at which tenant scope is injected, and the runtime hooks for recall and write across A2A delegation (with the grouping semantics themselves decided in component 5). Source surfaces: `pydantic-ai-server/pais/memory.py`, `pydantic-ai-server/pais/server.py` (`_create_memory`), `pydantic-ai-server/pais/serverutils.py`.
- **Key decisions.** Whether the tiers are one interface or composed interfaces; the long-term operation set and its synchronous-versus-asynchronous shape; how the contract stays uniform whether a single engine spans both tiers or the tiers are split across implementations; how recalled context is presented to the agent; whether the existing backends and `MEMORY_*` configuration are preserved or redesigned (resolved by [adr_0003](./adr_0003_memory-interface-and-runtime-data-plane.md) as a clean alpha redesign, not a compatible extension); where the OpenTelemetry boundary sits.
- **Primary inputs.** [KAOS-R7](../research/KAOS-R7-target-picture.md) data plane; [KAOS-R1](../research/KAOS-R1-memory-features-and-limitations.md) current interface; [KAOS-R5-5](../research/KAOS-R5-5-redis-agent-memory-server.md) two-tier model.
- **Dependencies.** Realises the model (component 1) through the engine (component 2) in the runtime, and is the contract the topology and control plane (component 4) operate and that tenancy and grouping (component 5) constrain. It deliberately follows components 1 and 2 because it dives deeper into implementation than either; it may later merge with one of them if the overlap proves large, but is kept separate for now.

### 4. Deployment topology, execution model, and control plane

This component merges the runtime deployment concerns with the declarative control plane, because the two are tightly coupled: the operator must deploy and operate exactly the topology the execution model chooses, and the CRD must describe it. Splitting them would force the same decisions to be made twice.

- **Scope.** How the long-term engine is run and how its work is scheduled — central shared service versus embedded library and the default for a multi-tenant fleet; the split between the synchronous, low-latency recall hot path and background extraction/consolidation; the background job system (queue and workers); high availability, resilience, and blast-radius handling (timeouts, circuit breaking, graceful degradation to working memory, caching, replicas and disruption budgets); and the engineering needed to make a stateful engine server horizontally scalable. Together with the control plane that operates it — the new `MemoryStore` (or similarly named) CRD describing a long-term backend by capability and policy, the slim memory block the Agent CRD retains to select a store and set per-agent scope and budget, the operator's reconciliation responsibilities (deploying the memory-service and worker workloads, wiring credentials as secrets, scheduling consolidation/forgetting jobs, exposing health and metrics, validating model references), datastore binding via connection and secret references rather than lifecycle ownership, polyglot store selection per tier, and ownership of schema versioning and migration. Source surfaces: `operator/api/v1alpha1/agent_types.go`, `operator/controllers/agent_controller.go`.
- **Key decisions.** The default topology and when the embedded alternative applies; the background execution mechanism (reuse of the existing Redis-backed queue versus a dedicated worker workload); the degradation contract when memory is unavailable; how the Mem0 server's singleton instance and per-container history store are made HA-safe ([KAOS-R5-1](../research/KAOS-R5-1-mem0.md)); per-tenant rate and quota enforcement on the shared service; the CRD schema and its capability-versus-vendor framing; the relationship between the Agent CRD and the `MemoryStore`; precisely what the operator deploys versus binds (reaffirming that underlying databases are externally operated, bring-your-own dependencies); the mapping of storage tiers to concrete stores (vector, optional graph, relational, Redis); and where schema migration and versioning live.
- **Primary inputs.** [KAOS-R7](../research/KAOS-R7-target-picture.md) deployment-topology, control-plane, and storage/distribution sections; [KAOS-R5-1](../research/KAOS-R5-1-mem0.md) statefulness caveats; [KAOS-R1](../research/KAOS-R1-memory-features-and-limitations.md) current control plane and autonomous-loop event volume; [KAOS-R6](../research/KAOS-R6-tool-selection.md) engine-agnostic direction.
- **Dependencies.** Depends on components 2 and 3; carries the grouping and governance fields decided in component 5.

### 5. Multi-tenancy, agent grouping, and governance

This component is where KAOS-specific grouping — including A2A fleets — is decided, building on the scope-addressing model defined in component 1. It is broader than generic multi-tenancy because KAOS agents are first-class resources that delegate to one another, so "who can see which memory" is both a tenancy question and an agent-topology question.

- **Scope.** How memory is shared or isolated across users, agents, tenants, and A2A-delegating fleets, given the scope-addressing model and its delegation-propagation rules from component 1; the relationship between Kubernetes resources and memory groupings versus a configuration-based grouping; the isolation modes spanning shared-store-with-scope-filtering through store-per-tenant; authentication and authorization on the always-on `/memory/*` endpoints; and the governance feature set — encryption at rest, per-tenant quotas, audit logging, export/import and backup, schema migration, and right-to-erasure deletion. KAOS surfaces for the A2A and delegation model: `pydantic-ai-server/pais/serverutils.py` (RemoteAgent), `pydantic-ai-server/pais/tools.py` (DelegationToolset).
- **Key decisions.**
  - **CRD relationships versus config-based grouping.** Whether agent-to-memory groupings — including delegated A2A sub-agents — are expressed through Kubernetes resource relationships (an Agent referencing a shared `MemoryStore`, owner references, or label selectors that define a group) or through a config-based scope path carried per agent (the segmented path defined in component 1), or a combination, and how an A2A sub-agent is placed into a group or scope by binding the delegation-propagation rules from component 1 to KAOS's A2A delegation surfaces (RemoteAgent, DelegationToolset). This is the central KAOS-native grouping decision, feeds the CRD design in component 4, and realises the cross-agent/organizational memory capability [KAOS-R7](../research/KAOS-R7-target-picture.md) names as under-served by per-user-centric tools.
  - **Isolation, access, and governance.** The enforced isolation model and how scope filtering is made non-optional at the data layer; the authentication mechanism for memory endpoints; which governance features are core versus deferred; and how erasure and export propagate across tiers and through the underlying engine.
- **Primary inputs.** [KAOS-R7](../research/KAOS-R7-target-picture.md) multi-tenancy, observability/governance, and capabilities-no-tool-provides sections; [KAOS-R1](../research/KAOS-R1-memory-features-and-limitations.md) governance gaps and the A2A/delegation model; [KAOS-R2](../research/KAOS-R2-memory-ecosystem-research.md) scoping model; the isolation findings across [KAOS-R5-1](../research/KAOS-R5-1-mem0.md) through [KAOS-R5-5](../research/KAOS-R5-5-redis-agent-memory-server.md).
- **Dependencies.** Builds on the scope-addressing model defined in component 1; cross-cuts the runtime interface (component 3, where scope is injected) and the control plane (component 4, where groupings are expressed).

## Cross-cutting concerns

These are not separate components; each ADR addresses them within its own scope rather than deferring them to a single record.

- **Observability.** OpenTelemetry instrumentation of every memory operation, emitted by KAOS regardless of the engine's native observability maturity ([KAOS-R7](../research/KAOS-R7-target-picture.md) observability section). The interface ADR (component 3) fixes the instrumentation boundary; each subsequent ADR states the spans and metrics its surface emits. If this grows beyond a cross-cutting concern it may be promoted to its own ADR.
- **Backward compatibility and migration.** This framing was overturned during the decision phase: because KAOS is in alpha, [adr_0003](./adr_0003_memory-interface-and-runtime-data-plane.md) and [adr_0004](./adr_0004_deployment-topology-and-control-plane.md) elect a clean break — the `Memory` method surface, the `MEMORY_*` configuration, and the inline `MemoryConfig` CRD block are redesigned rather than preserved, and deployments are migrated rather than transparently upgraded. The long-term tier remains *additive at the feature level* (an agent without a configured store is unaffected), but working-tier internals and configuration are not held compatible. Each ADR states its compatibility impact.
- **Security defaults.** Secret-based credentials, no operator-wide datastore default, and least-privilege scope are assumed by every component, not only component 5.

## Dependency and sequencing

The components form a dependency chain rather than a flat list, but they are authored in the order that lets each decision rest on a settled foundation. Component 1 (the memory model and operations) defines what memory means, what must be stored and recalled, and the structure of the scope dimension; component 2 (the memory implementation) chooses which engine backs which tiers and what is built versus adopted; component 3 (the runtime interface) realises the model through that implementation in the data plane and dives into the implementation contract; component 4 (topology, execution, and control plane) decides how it runs and how the operator makes it declarative and operable; and component 5 (multi-tenancy, agent grouping, and governance) cross-cuts the data and control planes, building on the scope model from component 1. The recommended authoring order is therefore 1, 2, 3, 4, 5 — with the cross-cutting concerns settled inside each.

## ADR conventions

- **Filename.** ADRs use `adr_NNNN_<slug>.md` with a zero-padded sequence number and a short kebab-case slug (for example `adr_0001_memory-model-and-lifecycle-operations.md`). This index uses the descriptive name `adr_high_level_components.md` and is not itself a numbered decision.
- **Numbering.** Numbers are assigned in authoring order and are provisional until an ADR is written; additional ADRs may slot in as decisions are split out. The numbering reflects the order decisions are recorded, not a strict mapping to the five components.
- **Status lifecycle.** Each ADR carries a status: `Proposed`, `Accepted`, `Superseded` (by a named later ADR), or `Deprecated`. Superseding never edits the original record; it adds a new one and updates the status.
- **Structure.** Each ADR follows a consistent shape: title and status; context (the problem and the relevant research, cited); the decision; consequences (positive, negative, and follow-on work); and alternatives considered with the reason each was not chosen. References link back to the research documents rather than restating them.
- **Authoring style.** ADRs follow the repository [CONVENTIONS](../CONVENTIONS.md): single-line paragraphs and list items, Markdown links for cross-references, repository-relative paths for KAOS source, and no references to phases, tasks, or plan steps in the document body.
