# KAOS-R11 — Short-term memory storage: why working memory stays a plain buffer

This document records the investigation into whether Mem0 can serve as the backend for short-term (working) memory as well as long-term memory, and how the datastore should be consolidated. It supports the runtime data-plane decision ([adr_high_level_components](../adrs/adr_high_level_components.md) component 3, ADR 0003) and refines the storage posture of [ADR 0002](../adrs/adr_0002_memory-implementation-mem0-and-pydantic-ai-integration.md). It exists because the engine choice raised a natural simplification — let Mem0 own both tiers so there is a single backend — and that simplification needed to be tested against Mem0's actual architecture rather than assumed. Findings are drawn from Mem0 source, official documentation, and GitHub issues, with citations.

## Question

Mem0 was selected as the long-term engine. Working memory today is a separate KAOS-owned tier (Local, Redis, Null backends). The tempting consolidation is to delegate working memory to Mem0 too, so the platform runs one memory system. The investigation asked: can Mem0 store raw conversation turns without the cost of a vector store, does it even have a working-memory concept, what do comparable frameworks do, and is pgvector or Redis the better single datastore.

## The embedding-call trade-off

Mem0 is architecturally a vector store plus an LLM extraction engine; there is no code path that persists a message without first embedding it. Storing anything requires turning the text into a vector through an embedding model — a separate model call with latency and per-token cost. With `infer=False` (extraction disabled), Mem0's `_add_to_vector_store` still calls `embedding_model.embed(msg_content)` once per non-system message, so every appended turn incurs an embedding call. With `infer=True` it instead embeds the concatenated conversation window once for deduplication, runs an extraction LLM call, and batch-embeds the extracted facts. Either way, putting working-memory turns into Mem0 pays a model call per turn or per window for vectors that working-memory retrieval never searches by similarity. By contrast, a plain relational row is a bare insert with no model call, retrieved by `session` and `created_at`. Source: `mem0/memory/main.py` `_add_to_vector_store`.

For long, busy conversations the window-embedding path is also fragile: an open pull request truncates the Phase 1 embedding input because conversations over roughly 8K tokens crash the embedding call with a token-limit error (`mem0ai/mem0` PR #5685). A plain buffer has no such ceiling.

## Mem0 has no usable working-memory primitive

Mem0's documentation describes short-term layers (conversation, session) alongside long-term (user, org), but the implementation maps every layer into the same vector store, differentiated only by scope filters. The only true raw-message buffer in Mem0 is a private SQLite `messages` table used to give the extraction LLM the last few turns of context: it is not exposed through any public `get_messages` API, it is capped at ten messages per scope, and it is process-local, so it is neither queryable nor safe across replicas in a multi-pod deployment. On the read path, `get_all(filters={"run_id": ...})` is a metadata filter scan rather than a similarity search, but it returns extracted facts (or, under `infer=False`, embedded messages), not the original turns in order — and on the pgvector backend `list()` has no `ORDER BY`, so it does not even return them in recency order. Source: `mem0/memory/storage.py` `save_messages`/`get_last_messages`; `mem0/vector_stores/pgvector.py` `list`; Mem0 docs `core-concepts/memory-types`.

The practical conclusion is that Mem0 does not provide a conversation buffer that KAOS can rely on; the buffer is expected to be owned by the calling framework, which is exactly KAOS's existing working-memory tier.

## What comparable frameworks do

Across the ecosystem, short-term memory is kept as a plain, non-embedded buffer and embeddings are reserved for long-term semantic recall. LangChain's buffer, token-buffer, and summary-buffer memories hold plain message lists, and LangGraph checkpointers serialize raw JSON state retrieved by thread id. LlamaIndex's chat memory buffer is a plain dictionary store trimmed by token count. Letta/MemGPT keeps recent messages in a PostgreSQL recall table and reserves the vector-embedded archival store for long-term. Google ADK separates an in-memory or database session service from a vector memory bank, and AutoGen separates a buffered chat context from a Chroma vector store. CrewAI is the partial exception: it uses a vector store for short-term but distinguishes an ordered `list_records` path from the embedded `search` path. Mem0 is the lone outlier that embeds everything regardless of tier. Sources: framework source and docs as cited in the comparison; LlamaIndex `chat_memory_buffer.py`, CrewAI `memory/storage/backend.py`.

## Corroborating GitHub signals

Issues on the Mem0 repository reinforce that the vector store is the wrong home for conversation history. A long-running report describes session-scoped memory returning old contradictory facts by similarity rather than recency, with the maintainers closing the underlying semantic-conflict requests as not planned (issue #5352, and #4896/#4904 closed as not planned). A high-priority issue reports concurrent asynchronous writes corrupting the vector index, which is precisely the high-frequency write pattern that per-turn working-memory storage would create (issue #4892). Another reports entity embeddings merging across distinct same-named entities, a cross-tenant contamination risk for a multi-tenant platform (issue #5438). None of these affect Mem0's intended long-term use; they only bite if Mem0 is pushed into per-turn short-term storage.

## pgvector versus Redis as the single datastore

For a durable, multi-tenant, Kubernetes-hosted system of record, pgvector is the better default by a wide margin. It is ACID and WAL-durable, uses HNSW or DiskANN approximate indexes that scale beyond brute force, supports rich JSONB filter operators and native full-text indexing, needs only the standard `vector` extension (available on common managed Postgres offerings), and ships connection pooling suited to multi-replica clients. Mem0's Redis backend requires the Redis Stack / RediSearch module rather than plain Redis OSS, uses a brute-force flat vector index, supports only tag-equality filtering, holds the dataset in RAM, and is volatile unless persistence is configured. Redis's one genuine advantage is recency-ordered `list()` and raw write throughput; the recency gap on pgvector is a small, fixable omission of an `ORDER BY`. Source: `mem0/vector_stores/redis.py`, `mem0/vector_stores/pgvector.py`.

## Conclusion and recommendation

Short-term memory stays a KAOS-owned plain buffer; it is not delegated to Mem0. The datastore is consolidated to a single Postgres instance: working memory as a plain relational table (session, role, content, created_at — no embeddings, recency-ordered, token-windowed in KAOS), and Mem0 long-term as pgvector rows in the same instance. This meets the real goal of operating one datastore and dropping Redis, without forcing a vector engine to do a recency job it does poorly and expensively. Delegating both tiers to Mem0 was rejected on per-turn embedding cost and latency, the absence of a public ordered raw-turn store, a process-local non-replica-safe internal buffer, a long-conversation embedding ceiling, and concurrency and isolation issues that only appear under high-frequency short-term writes.

## Caveats

Source offsets and issue states are as of late June 2026 and Mem0's API is actively changing. The Letta/MemGPT and Google ADK findings were taken from documentation rather than verified from source. The pgvector recency-ordering gap could be closed upstream, which would narrow but not overturn the recommendation, since the embedding-per-turn cost is the decisive factor.
