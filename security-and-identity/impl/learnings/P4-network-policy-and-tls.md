# P4 — network policy and TLS (learnings)

## What worked

- **Splitting the bypass problem into "block" + "reroute"** was the key insight that the post-P10 review surfaced. NetworkPolicy alone only *denies* direct traffic; it does not redirect it. Because the operator injects direct `svc.cluster.local` URLs for every internal hop, enabling NetworkPolicy without also repointing those URLs would have broken agent→MCP/ModelAPI/peer traffic outright. Treating them as two halves of one feature (`security.networkPolicy.enabled` to block, `security.gatewayRouting.enabled` to reroute) made the design correct and each half independently testable.
- **The pre-existing HTTPRoutes were already shaped for repointing.** All three resource types had routes with path-prefix `/{ns}/{type}/{name}` + URLRewrite `ReplacePrefixMatch: "/"` and *no* hostname binding. So repointing a client URL to `http://<gatewayHost>/<ns>/<type>/<name>` works with zero route changes: the client appends its suffix, the gateway strips the prefix, and the backend receives the original path. `gateway.GatewayEndpoint(...)` already built exactly this string but was unused — P4 just wired it in.
- **Mutating the in-memory endpoint values inside the agent controller** (rewriting `modelapi.Status.Endpoint` and the mcp/peer map values before the env block is built) kept the change local and side-effect-free: the agent controller only *reads* those endpoints to build env, it never writes them back to the API server, so the repoint cannot leak into another reconciler.
- **Ingress-only NetworkPolicy keyed on namespace labels** (allow-from gateway ns + operator ns) is the minimal policy that forces gateway-only reachability. Deduping the two `from` peers when the gateway and operator share a namespace keeps the rendered policy clean.
- **Lookup-guarded `genSelfSignedCert`** for the selfSigned TLS mode means a `helm upgrade` reuses the existing cert (read back via `lookup`) instead of regenerating it every apply, avoiding churn and pod restarts.

## Gotchas / decisions

- **Gateway host resolution must fail safe.** Order: explicit `SECURITY_GATEWAY_HOST` → Gateway `.status.addresses[0].value` (the MetalLB/LoadBalancer IP, in-cluster reachable on KIND) → fall back to direct URLs. Falling back to direct URLs rather than to an empty/broken host means a misconfigured gateway host degrades to the pre-P4 behaviour instead of breaking all agent traffic.
- **Gateway routing is default-off, NetworkPolicy default-on (when auth is operational).** This is intentional: an install that hasn't opted into gateway-enforced authz should not have its traffic paths silently rerouted. But the two are only *useful together* — NetworkPolicy on + routing off would block legitimate traffic. The CLI/chart expose both so an operator enables them as a pair; this pairing requirement is documented rather than auto-coupled, to keep each flag's meaning orthogonal.
- **kindnet does not enforce NetworkPolicy.** The default KIND CNI accepts the objects but never drops traffic, so the local `kind-kaos-e2e` cluster cannot prove enforcement. The objects are always generated; genuine enforcement (direct curl denied, gateway path allowed) requires a Calico/Cilium cluster and is validated in the end-to-end block, not in P4's branch CI.
- **`make manifests` is safe here but `make helm` is not.** NetworkPolicy is a built-in type, so regenerating `config/rbac/role.yaml` for the new `networkpolicies` rule introduces no CRD drift. The Helm chart RBAC (`kaos-operator-rbac.yaml`) is hand-maintained and was mirrored by hand — `make helm` would clobber the hand-maintained chart and must not be run.
- **`make test-unit` fails locally** on a go1.26-vs-1.25 coverage toolchain mismatch; `go test ./pkg/... ./controllers/...` is the reliable local invocation.

## Carry-forward

- The **genuine in-cluster enforcement e2e** (Calico KIND, direct ClusterIP curl denied, repointed gateway path allowed + authz-checked, HTTPS terminated, full P6–P10 authz dataflow intact under enforcement) runs in the dedicated end-to-end block that follows P4. That block is where the NetworkPolicy `from` peers must be proven to allow *every* required auth dataflow (agent→gateway, gateway→ext_authz/ext_proc, gateway→broker token+JWKS, agent→broker mint+exchange, sync→broker admin, gateway→Keycloak/AIB JWKS) — an over-tight policy would surface there as a broken authz path.
- **Egress NetworkPolicy and TLS hardening are P12's job.** P4 ships ingress-only policies and gateway HTTPS termination; locking down egress (so a compromised pod cannot exfiltrate) and enforcing TLS on internal hops belong to the enforcement/hardening phase.
- When gateway routing is enabled, the gateway host should ideally be a stable in-cluster DNS name rather than a LoadBalancer IP; the status-address fallback is convenient for KIND but a production install should set `--gateway-host` explicitly.
