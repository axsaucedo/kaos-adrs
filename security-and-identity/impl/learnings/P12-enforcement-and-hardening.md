# P12 — Enforcement and operational hardening (learnings)

## Detect gateway outcomes from response headers, never from the body or status code alone

The runtime's outbound path is instrumented at the `httpx` transport, so it sees raw responses from MCP/ModelAPI/peer calls that may have been short-circuited by the gateway's ext_authz (denied, 403) or ext_proc (re-auth, an HTTP 200 JSON-RPC error). Keying outcome detection on the status code is wrong: the ext_proc re-auth immediate response is a 200, and a 403 from an upstream service is not necessarily a KAOS authorization denial. Keying on the body is also wrong: it forces parsing every response and couples the SDK to a JSON-RPC shape. The robust contract is a dedicated response header — `x-kaos-access-reason` (presence = this is a KAOS gateway outcome, any status) and `x-kaos-reauth-url` (presence promotes the outcome to re-authentication with that URL). This makes detection cheap, body-agnostic, and unambiguous, and it is why the AIB ext_proc immediate response was extended to set the headers in addition to the JSON-RPC `data` fields.

## Avoid the circular import by putting the header constants where the lower module lives

`aib/client.py` already imports `ctx` from `aib/instrument.py`, so the header-name constants (`HEADER_ACCESS_REASON` / `HEADER_REAUTH_URL`) must live in `instrument.py` and be re-imported by `client.py` — not the other way round. The instrument module's outcome helper then lazy-imports `client.raise_for_gateway_outcome` only when the header is actually present on a response, so the common no-security path pays nothing and there is no import cycle.

## Shared outcome helpers cannot live in server.py — a2a.py imports server, not vice versa

`pais/server.py` imports `setup_a2a_routes` from `pais/a2a.py`, so `a2a.py` cannot import from `server.py`. The synchronous path (server) and the autonomous path (a2a) both need to recognise an access outcome in an exception chain and format it, so those helpers went into a new leaf module `pais/outcomes.py` (`find_access_outcome` walks `__cause__`/`__context__`, `format_access_outcome`, `access_event_data`). Putting cross-cutting helpers in a dependency-free leaf module is the clean way to share logic between two modules that already have a one-directional import edge.

## In the autonomous loop, record one event and stop — do not continue

When an access outcome is raised inside the autonomous iteration's `try/except`, the wrong reflex is to `continue` so the agent "keeps trying". That re-hits the same denial every iteration — a retry storm that never resolves, because the missing grant/re-auth is a human action, not a transient fault. The correct behaviour is to record exactly one structured `user_action_required` event (plus `task.metadata`), set the last response, and `break` — ending the run gracefully so the post-loop completion path marks the task done. "Exactly one event, then end" satisfies both the autonomous and async-task modes without hanging and without auto-granting.

## The current bootstrap encoding auto-grants from CRD references — and that is the honest, documented limitation

The crux of the requested-vs-approved work: AIB's access-check `Decide` allows whenever a **bound** permission set covers the target service+action (`covers()` checks scope only, with no approval/status filter), and the sync projection **binds** every projected permission set to the agent. So today a CRD reference → sync binds permission set → access-check allows — i.e. the reference *is* the grant. This is the temporary bootstrap encoding (the service comment calls platform resource grants "temporarily encoded as synthetic services plus PermissionSets"). A first-class requested-vs-approved model (distinct `RequestedAccessEdge` / approved `ResourceGrant` with a real approval gate) is explicitly out of scope here and tracked for a later phase. The in-scope, honest deliverable was therefore to *guard and document* rather than to build the gate: lock the invariant that the projection never emits an approval status of its own (so the binding stays the single, clearly-temporary grant representation), confirm the operator writes no approval state, and keep requested edges (`granted_resources`) structurally separate from the binding so the gate can be inserted later without restructuring. The full deny-until-approved end-to-end proof depends on that future model; the deny path itself is already exercised whenever the binding/permission-set is absent.

## Backend-neutrality is a configuration property, and the cheapest proof is a table test plus a policy eval

ADR-005 claims the ext_authz integration is backend-neutral — OPA via opa-envoy is a drop-in over the same `envoy.service.auth.v3.Authorization/Check` contract with neutral `kaos.resource`/`kaos.action` context extensions. The operator already generates the `SecurityPolicy.spec.extAuth.grpc.backendRef` from a single config value (`ExtAuthzURL`), so the swap genuinely is config-only. Proving it does not require standing up OPA in CI: a table test asserting the generated backend ref follows the configured URL for both the AIB and opa-envoy hosts locks the "no code change" half, and an `opa eval` of the sample Rego (allow for a granted edge, deny for an ungranted one, decoding the actor `sub` from the bearer token and matching `kaos.resource`/`kaos.action` against a grant bundle) locks the "equivalent decision" half. Together they demonstrate the property cheaply and deterministically; the live KIND swap is then a confirmation, not the evidence.

## opa-envoy reads the decision from a configured Rego path and packages a pre-built binary like AIB

Two operational gotchas for the OPA drop-in. First, opa-envoy decides from `plugins.envoy_ext_authz_grpc.path` (set to `kaos/authz/allow` here) — the policy package/rule name must match that path exactly or every request denies. Second, the same Dockerfile lesson from P11 applies to any image whose entrypoint loads files: mount the Rego + data via a ConfigMap (as the sample does) so the policy is updatable without rebuilding the image. The `import rego.v1` form requires the `if`/`in` keywords and `count("Bearer ")` to strip the bearer prefix before `io.jwt.decode`, which decodes (does not verify) the already-gateway-validated token.

## Keep every new enforcement path default-off and additive

The through-line of P12: each new behaviour is gated so a security-default-off install is byte-for-byte unchanged. Egress NetworkPolicy is gated by `SECURITY_NETWORK_POLICY_EGRESS_ENABLED` (default off), the runtime outcome surfacing only triggers on the presence of a `x-kaos-access-reason` header that a non-secured gateway never sets, the `user_action_required` event only appears when an outcome is actually raised, and the OPA backend is a demonstration with AIB remaining the default. This is what makes a cross-cutting enforcement phase safe to merge: the secured posture gets materially stronger while the unsecured default is provably untouched, which is also exactly what the regression check (security-off path unchanged) asserts.
