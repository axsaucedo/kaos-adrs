# OPA-in-ExtProc authorization — model parity, autonomous gap, and OAuth path (learnings)

**Date**: 2026-07-07
**Context**: AIB maintainers moved authorization from a standalone Envoy `ext_authz` decision service (the KAOS-authored `access-check` on AIB `#398`) into OPA embedded in the ExtProc token-exchange binary (AIB `#222`, merged to AIB `main` as `feat(020)`, commit `9ce66afb`). The KAOS-authored `#398` (standalone ext_authz) and `#399` (python SDK) branches are abandoned and treated as gone. KAOS now targets AIB **`main` (incl. `#222`) + `#397`** (deployability foundation) as the initial-release surface. These learnings capture what the OPA-in-ExtProc model can and cannot enforce, verified against the merged `#222` code and the KAOS actor-authz experiment (worktree `../aib-222-verify`, untracked fixtures under `tests/e2e/extproc/fixtures/policies/`).

## Headline

The distinction we have been calling "Model 1" (KAOS-owned OPA data) versus "Model 2" (broker permission-sets) is **only about where the authorization facts come from, not about which identities can be enforced**. Both feed the same OPA evaluation and both can key on the user subject *and* the calling agent actor *and* the target resource. There is full identity parity between the two models. The one hard constraint — a request with no user subject bearer bypasses OPA entirely — applies equally to both.

## The OPA input exposes four fact sources simultaneously

`#222`'s `internal/extproc/authorization/input_builder.go` builds an ext_authz-compatible input and passes **all** HTTP headers through verbatim (`buildCheckRequest` copies the header map into `request.http.headers`). A single Rego policy in package `aib.extproc.authz` (decision document `result`) therefore sees, in one evaluation:

- **Subject (user) identity** — decoded in-policy from the `authorization` header (the Keycloak/OIDC user token).
- **Actor (agent) identity** — decoded in-policy from the `x-agent-authorization` header (the AIB-signed agent actor token).
- **KAOS-owned data** — `data.kaos.grants` (or any shape we choose), a static data document generated by KAOS from the Agent/MCPServer/ModelAPI CRDs. This is the "Model 1" fact source.
- **Broker granted permission sets** — `context.granted_permission_sets`, returned by the RFC 8693 token-exchange and injected by ExtProc. This is the "Model 2" fact source.

Because all four are present in the same input document, the model choice is purely a **data-provenance** decision. Both models can express actor-only, subject-only, or combined `(subject, actor, resource)` rules.

## Both models can answer "can user A reach resource X via agent B?"

The KAOS experiment's `kaos_actor.rego` and its data-driven variant `kaos_bundle/kaos_actor_data.rego` keyed on the **actor** only (`data.kaos.grants[actor_sub]` where `actor_sub` is the `sub` of the actor token). That was the axis under test, not a limitation of the mechanism. Extending to a subject-aware rule — for example keying on `(subject, actor)` or intersecting a per-user allowed set with a per-agent allowed set — is a **data-shape and Rego change, not a different mechanism**, because the subject token is already in the input. Consequently we can make **both models use the identical approach** (OPA reads the input; KAOS decides the grant-data shape) and both can enforce the delegated "user A → resource X via agent B" question.

## The one hard constraint (both models): no user subject bearer ⇒ OPA is bypassed

`#222` gates OPA behind the presence of a subject bearer. In `internal/extproc/server/server.go`, `processRequestHeadersOPA` extracts the bearer from the `authorization` header and, when it is empty, returns `passThrough()` **without invoking the authorizer at all** (server.go:407-409). The KAOS experiment encodes this as an explicit test (`tests/e2e/extproc/opa_kaos_actor_test.go:118-138`): a request carrying a valid `x-agent-authorization` actor token but **no** `Authorization` bearer bypasses OPA entirely — the actor→resource policy is never consulted, even though it would have denied.

Consequences:

- **User-present flows always enforce.** "User A via agent B" carries a user subject token, so OPA runs and both models apply. This is the common interactive case and is fully covered.
- **Pure-autonomous agent→resource does not enforce.** An autonomous run with only an actor token (no user subject) passes through with no authorization decision. This is the previously identified **G1 autonomous gap**, and it is a property of `#222`, independent of which model supplies the grant data.
- **`B-i` (defer autonomous as unsupported) therefore means autonomous runs get no gateway authorization at all**, not merely "no third-party token." Closing the gap would require either an AIB change to also evaluate OPA on actor-only requests, or `B-ii` (treat the agent actor token as the bearer in autonomous mode), which collapses the two-identity model and is out of scope for now.

## Evaluation ordering relative to token exchange

`#222` runs authorization and token exchange in a specific order that matters for both models (server.go doc comment lines 4-5, and `processRequestHeadersOPA` / `processRequestBody`):

- **Body-bearing requests** (e.g. MCP JSON-RPC `POST`): token exchange runs eagerly in the `RequestHeaders` phase, its `granted_permission_sets` are stashed in per-stream state, the Authorization header is replaced with the exchanged token, the body is buffered, and OPA evaluates in the `RequestBody` phase with both the body and `context.granted_permission_sets` available (server.go:469-476, then `BuildOPAInput`).
- **Header-only requests** (e.g. SSE `GET /mcp`): exchange is deferred; OPA evaluates first on headers only, and `context.granted_permission_sets` is unavailable by design (`BuildOPAInputHeadersOnly`).

This means that on the body-bearing path a token exchange happens **before** OPA even for a pure "Model 1" internal decision that conceptually needs only the actor token and KAOS data. Whether that mandatory exchange is benign for an internal resource with no third-party mapping (does the broker return a usable re-scoped token, or fail closed?) is an **open implementation point to validate live** before relying on Model 1 for internal agent→MCP calls; it does not change the model design.

## Model 2 OAuth path (we are not diverging from standard OAuth)

The "Model 2" path reuses mechanisms already on AIB `main`; there is no new or divergent OAuth flow:

1. The user authenticates to Keycloak (OIDC) and obtains a subject token carried in `Authorization: Bearer`.
2. The agent authenticates to AIB via a `client_credentials` grant and obtains its actor token, carried in `x-agent-authorization`.
3. At the gateway, ExtProc performs an RFC 8693 token exchange using the subject token. The exchange returns `granted_permission_sets` for the authorization decision, and — **only for external third-party targets** — a vaulted user OAuth token (from AIB's consent/third-party-session store) that replaces the upstream `Authorization` header.

For **internal KAOS resources** (agent→MCP, agent→ModelAPI, agent→agent) the exchange yields the granted sets for an allow/deny decision and no third-party vault token is involved. For **external services** (GitHub, Google, …) the exchange additionally provides the delegated third-party token via AIB's existing consent + token-vault feature, unchanged. In short, Model 2's OAuth path is Keycloak (user) + `client_credentials` (agent) + RFC 8693 exchange (gateway) — all pre-existing.

## Net design implication

- Pick the models by **data provenance and lifecycle ownership**, not by "what they can enforce" — both can enforce the full `(subject, actor, resource)` space.
- Design the KAOS Rego to read whichever fact sources are enabled, so a single policy can consult `data.kaos.grants` and `context.granted_permission_sets` together.
- Accept the autonomous gap (`G1`) as a shared `#222` property; document autonomous authorization as unsupported until an AIB-side change or an explicit actor-as-bearer decision is made.
