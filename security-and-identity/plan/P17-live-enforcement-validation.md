## Problem statement

P16 collapsed the install surface into two presets and validated them at the unit, chart-render and CI levels, plus the `kaos-internal-demo` posture's projection path via the operator e2e. Two validations planned for P16 could not be executed live and are carried here, together with the one real feature gap they depend on:

1. **The `keycloak-aib-enabled` posture does not yet enable the broker ext_proc OPA authorization decision.** The install wires token exchange and per-agent credentials, and the operator registers services/permission-sets/agents into the broker, but nothing sets `EXTPROC_AUTHORIZATION_*` or mounts a rego reading `granted_permission_sets`. So a valid broker-issued actor token cannot produce a real allow/deny on the gateway-routed path — the verified posture enforces JWT presence but not the authorization graph. This is a genuine wiring gap, not just a missing test.
2. **Strict gateway-only enforcement has never been proven live.** The generated bypass-prevention NetworkPolicies are only enforced by a CNI that implements the NetworkPolicy API. The default KIND CNI (kindnet) accepts NetworkPolicy objects but never enforces them, so a "direct ClusterIP bypass is blocked" assertion passes for the wrong reason. Real enforcement requires a Calico KIND cluster (`disableDefaultCNI: true`).

Closing these makes the verified posture actually authorize, and turns strict gateway-only traffic from "generated" into "proven".

## Current-state grounding (researched)

- **AIB ext_proc OPA capability (confirmed in the binary)**: `internal/extproc/authorization/authorizer.go` loads a rego from `authorization.policy.path` (file/dir, compile-once) or `authorization.policy.config_file` (bundle/SDK). The loader binds env with prefix `EXTPROC` + `.`→`_` (`EXTPROC_AUTHORIZATION_ENABLED`, `EXTPROC_AUTHORIZATION_POLICY_PATH`, `EXTPROC_AUTHORIZATION_POLICY_PACKAGE`, `EXTPROC_AUTHORIZATION_POLICY_DECISION`). The AIB chart in the working worktree templates only the token-exchange CEL authorization, not the ext_proc OPA block — so OPA authz must be delivered via env + a volume-mounted rego regardless of the chart.
- **`#222` structural constraints (learnings)**: OPA is skipped when no user subject bearer is present (the autonomous G1 gap, tracked as followup F3); exchange runs before OPA on body-bearing requests; the exchange is subject-only (azp-freeze). Keying the rego on the `x-agent-authorization` actor header side-steps azp for allow/deny, but the agent runtime must inject that header per hop and ext_proc must run in body mode.
- **Operator projection**: `aib` provider → `BrokerProjector` registers services/permission-sets/agents and mints per-agent credential Secrets; it emits no rego ConfigMap today. `kaos` provider → `ConfigMapProjector` already writes `policy.rego` (+ optional `data.json`), so the same rego content can seed the broker mount.
- **Enforcement CNI**: kindnet does not enforce NetworkPolicy; Calico does. Prior enforcement evidence (TEST A direct-ClusterIP BLOCKED / TEST B via-gateway 401) is captured in `impl/learnings/manual-e2e-P6-P10-validation.md` on a dedicated Calico cluster (`kaos-sec-e2e`, `disableDefaultCNI: true`, pod subnet `192.168.0.0/16`, Calico v3.27.3).
- **Preset wiring**: the `keycloak-aib-enabled` expander in `kaos-cli/kaos_cli/install.py` produces the AIB install args; extending it to emit the OPA-authz env + rego mount is the natural home for the fix.

## Design plan (how it fits)

### Broker OPA-authz enablement (feature)

Extend the `aib` install path so the `keycloak-aib-enabled` preset:
- Sets `EXTPROC_AUTHORIZATION_ENABLED=true` and the `EXTPROC_AUTHORIZATION_POLICY_{PATH,PACKAGE,DECISION}` values on the broker ext_proc deployment.
- Mounts a rego that reads `granted_permission_sets` and keys allow/deny on the `x-agent-authorization` actor header, delivered as a volume mount (chart does not template the OPA block). Reuse the same rego the operator's `ConfigMapProjector` already renders so the two providers share one policy shape.
- Confirms ext_proc runs in body mode so the actor header is available to OPA.

This is additive to the existing token-exchange wiring; the demo posture is unchanged.

### Live verified-posture allow/deny E2E (validation)

A reproducible bring-up that installs the `keycloak-aib-enabled` preset (Keycloak + broker + operator + agent + MCP), grants one edge, denies another, mints an actor token, and asserts a user-present allow next to a denied edge on the real gateway-routed path. Captured as a manual runbook mirroring the `manual-e2e-*` format, with the assertions a future automated tier should make.

### Live strict NetworkPolicy enforcement (validation)

Recreate KIND with Calico (`disableDefaultCNI: true`), install with `--gateway-api-strict`, and run the TEST A / TEST B pair: direct ClusterIP from a scratch pod must be BLOCKED; the same request via the gateway must reach the workload (subject to auth). Captured as a Calico-gated runbook; not runnable in kindnet CI.
