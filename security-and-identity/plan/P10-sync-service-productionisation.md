## Problem statement

P5 made the in-repo sync service correct and observable: it projects KAOS Agent/MCPServer/ModelAPI into AIB services/PermissionSets/agents/edges, mints per-agent client credentials into Secrets, prunes deleted records, reports drift, and exposes metrics + health/readiness with bounded retry/backoff. It is still a **single-replica, fixed-interval poll loop** that only *mints* credentials (never rotates them), reports status only through logs/metrics (nothing written back onto the KAOS resources), is installed exclusively from a **local chart path** (no published image/chart), and lives **inside the KAOS repo** with its release coupled to KAOS. P10 closes the productionisation gap defined by ADR-KAOS-008's "later — own repository" path and P5's explicit "Out of scope" list: (1) **watch-based incremental reconcile** (react to changes, not just poll), (2) **leader election** so the service can run as an HA Deployment without double-minting/double-pruning, (3) **active, time/policy-driven credential rotation** with a grace window, (4) **CRD status/annotation write-back** so operators see per-resource sync state on the KAOS objects, (5) **packaging** as a versioned image + published chart, and (6) **extraction** of the service into its own repository with its own release/CI, consumed by the CLI as a versioned external chart/image exactly as KAOS already consumes Keycloak and AIB. None of this re-does P5 work; it builds on the P5 end-state.

## Design plan (how it fits)

### Runtime: watch-based incremental reconcile (event-driven + periodic resync)

- Add a watch layer in `k8s.py` (`watch_resources(namespaces, resource_version, ...)` using `kubernetes.watch.Watch` over the three KAOS CRD list endpoints) that yields change events (ADDED/MODIFIED/DELETED) with the resource body and `resourceVersion`. In `main.py`, replace the bare sleep loop with a **debounced, event-driven reconcile**: maintain a "dirty" flag set by any watch event; a worker performs a full `run_once(...)` when dirty (coalescing bursts within a short debounce window, e.g. ~1–2s) **and** on a periodic safety-net resync (the existing `reconcile_interval_seconds`, repurposed as the max time between full passes). This keeps the simple, idempotent full-reconcile model (no per-object diffing — reconcile already converges) while reacting to changes in near-real-time and dramatically cutting idle API churn. Watch disconnects (410 Gone / expired `resourceVersion`) fall back to a full list+resync and re-establish the watch — never crash the loop. Keep it single-threaded-friendly (a simple thread for the watch feeding a flag/queue read by the reconcile worker), no async runtime needed.

### Runtime: leader election (HA-safe single active reconciler)

- Add Lease-based leader election (`coordination.k8s.io/v1` `Lease`) so the Deployment can run `replicas > 1` with exactly one **active** reconciler. Implement a small `LeaderElector` in `k8s.py` (acquire/renew a Lease named e.g. `kaos-sync-leader` in the service's namespace, holder identity = pod name from `POD_NAME`/hostname, configurable lease/renew/retry durations via `KAOS_SYNC_LEADER_*`). `main.py` only runs the watch+reconcile worker **while leader**; on lost leadership it stops reconciling/minting/pruning and stands by (re-contending). Default behaviour with a single replica is unchanged (it simply always wins the lease). Gate via a `KAOS_SYNC_LEADER_ELECTION_ENABLED` flag (default on; can be disabled for the simplest single-pod dev install). New RBAC: `coordination.k8s.io` `leases` create/get/update in the service namespace — add to the sync chart Role/RoleBinding.

### Runtime: active credential rotation (time/policy-driven, with grace)

- Extend reconcile so per-agent credentials are not just minted once but **rotated** on a policy: track each credential Secret's age (annotation `kaos.dev/aib-credential-rotated-at` written at mint/rotate time) and, when it exceeds `KAOS_SYNC_CREDENTIAL_ROTATION_SECONDS` (0 = disabled, the default-safe value until validated), call the AIB admin rotate/re-generate path to issue a new `client_secret` for the existing agent, update the Secret in place, and record the new rotation timestamp. Honour AIB's previous-secret **grace window** so in-flight agents using the old secret (and the P9 SDK file-reload) pick up the new one before the old is invalidated — rotate, write the Secret, then let AIB expire the old secret after its grace, never invalidate-then-write. Rotation runs inside the leader's reconcile pass and is idempotent (only rotates when due). This is the active counterpart to P5's mint-once behaviour and directly enables the P9 SDK rotation-reload path end-to-end.

### Runtime: CRD status / annotation write-back

- After each reconcile, write per-resource sync state back onto the KAOS objects as **annotations** (the ADR's "optional Kubernetes annotations/status integration", and the least-coupling option since the sync service is deliberately not an operator with CRD `.status` subresource ownership): on each Agent/MCPServer/ModelAPI, set `kaos.dev/aib-synced-at` (RFC3339), `kaos.dev/aib-external-id` (the projected `external_id`), and `kaos.dev/aib-sync-status` (`ok` / `drift` / `error` with a short reason in `kaos.dev/aib-sync-message`). Use a JSON-merge/strategic-merge **annotation patch** (additive, never touching `spec`) in `k8s.py`. Gate via `KAOS_SYNC_STATUS_ANNOTATIONS_ENABLED` (default on). New RBAC: `patch` on the three CRDs (the sync Role currently only needs get/list/watch) — add to the chart. This gives operators per-object visibility (`kubectl get agent -o jsonpath` / describe) without embedding sync into the operator.

### Packaging: versioned image + published chart

- Add a `kaos-sync` image to `release.yaml` `build-images` (matrix entry: context `sync-service`, image `axsauze/kaos-sync`, multi-arch like the others) and the corresponding `values.yaml`/chart image-tag `sed` updates in `build-helm`/`bump-version` so the sync image tag tracks the release version (same treatment as `kaos-agent`/`kaos-mcp-*`). Ensure `sync-service/chart/` references the published image (`axsauze/kaos-sync:<version>`) with a values override, and that the chart is packageable/versioned. Bump version files consistently with the rest of the release machinery (`sync-service/pyproject.toml` already at `0.4.8.dev0`; keep it in lockstep). This makes the service installable as a versioned image+chart rather than only a local path.

### Extraction: subtree mirror + standalone release + CLI consumption

- Add `sync-service` as a third subtree everywhere the pattern lives: a `sync-kaos-sync` job in `sync-repos.yaml` (`git subtree push --prefix=sync-service <kaos-sync-remote> main`, guarded on `CROSS_REPO_TOKEN`), a `check_subtree sync-service ...` entry in `subtree-check.yaml`, and a `release-standalone-repos` step creating the release on `axsaucedo/kaos-sync` (idempotent skip-if-exists). Repoint the CLI (`install.py`) so the **default** sync install consumes the published versioned chart/image (repo add + `helm upgrade --install --version <release>`), keeping `--sync-chart-path` as the documented dev/unpublished fallback (do not delete it). All new wiring is non-failing when the standalone repo/secret is absent, so this lands safely before the repo's first mirror. Update docs to describe the extracted-repo install path.
