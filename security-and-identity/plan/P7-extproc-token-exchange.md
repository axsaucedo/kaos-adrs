## Problem statement

Deliver delegated third-party tokens inline at the gateway. Today the gateway pipeline is `jwt_authn (P6) -> ext_authz (P2) -> upstream`: it authenticates the agent actor and the user subject, and authorizes the actor->resource edge, but it cannot turn a verified user subject token into a third-party access token for a protected downstream call. P7 adds the **`ext_proc` token-exchange** stage so that, when a call needs a delegated third-party token, the gateway calls AIB's ExtProc service which performs an RFC 8693 token exchange against the broker token endpoint and replaces the upstream `Authorization` header with the exchanged token — completing the pipeline `jwt_authn -> ext_authz -> ext_proc -> upstream`. Validation follows the proposed split: prove the path first with a **dummy third-party service + delegated grant**, then a real in-cluster delegated call. The CRD identity override (`spec.security.id`) is P8; per-tool/argument authorization stays out of scope.

## Current-state grounding (researched)

- **Gateway pipeline today**: the operator generates one Envoy Gateway `SecurityPolicy` per protected route (`operator/pkg/security/securitypolicy.go`) carrying `spec.extAuth` (ext_authz, P2) and — once P6 lands — `spec.jwt` (two `jwt_authn` providers). **No `ext_proc` anywhere.** `SecurityPolicy` is GVK `gateway.envoyproxy.io/v1alpha1` `SecurityPolicy`, built as an unstructured object; `ReconcileSecurityPolicy` create-or-updates it and is a no-op unless `cfg.IsOperational()`. It is called from all three controllers (`agent_controller.go:329`, `mcpserver_controller.go:222`, `modelapi_controller.go:296`) inside an `if secCfg := security.GetConfig(); secCfg.IsOperational()` guard, using `gateway.HTTPRouteName(ResourceType, name)` as both policy and route name.
- **Envoy Gateway `ext_proc` is a DIFFERENT CRD.** Envoy Gateway (pinned to `v1.4.6` in `kaos-cli/kaos_cli/install.py:19` `ENVOY_GATEWAY_VERSION`) configures External Processing via **`EnvoyExtensionPolicy`** (`gateway.envoyproxy.io/v1alpha1`, kind `EnvoyExtensionPolicy`), NOT `SecurityPolicy`. Its spec carries `targetRefs` (same HTTPRoute targeting shape as SecurityPolicy) and `spec.extProc: [ { backendRefs: [...], processingMode: {...} } ]`. **The exact field names/shape (`backendRefs` vs `backendRef`, the `processingMode.request`/`response` enum values, and whether a `failOpen`/failure-mode field exists at the policy level) are EG-version-specific and MUST be verified on the running v1.4.6 cluster before encoding** — do not assume; experiment under `./tmp/` first (`kubectl explain envoyextensionpolicy.spec.extProc`).
- **Operator security config** (`operator/pkg/security/config.go`): holds `ExtAuthzURL` (`SECURITY_AGENT_AUTH_EXT_AUTHZ_URL`), `Issuer`, `CredentialSecretPrefix` (+ P4/P6 fields once those land). `IsOperational()` keys on `ExtAuthzURL`. `ExtAuthzBackendRef()` parses a `host:port` DNS name into Service `name`/`namespace`/`port` for the gRPC backendRef. **No `ExtProcURL` / `SECURITY_AGENT_AUTH_EXT_PROC_URL`.** Env wired from `operator/chart/templates/operator-configmap.yaml`; RBAC for `gateway.envoyproxy.io/securitypolicies` is in `operator/config/rbac/role.yaml:51-62` and `operator/chart/templates/kaos-operator-rbac.yaml:65`.
- **AIB ExtProc service** (clone `agentic-identity-broker/`): a **separate deployable** (`Dockerfile.extproc`, compose service `extproc-token-exchange`/container `aib-extproc`), implementing `envoy.service.ext_proc.v3.ExternalProcessor` on gRPC **`:50051`** (`internal/extproc/server/server.go`, `internal/extproc/config/loader.go:54-56`). Config is `EXTPROC_*` (Viper/env): `EXTPROC_GRPC_BIND`/`_PORT`; `EXTPROC_OAUTH2_TOKEN_ENDPOINT` (AIB token endpoint), `EXTPROC_OAUTH2_ISSUER`, `EXTPROC_OAUTH2_CLIENT_ID`, `EXTPROC_OAUTH2_CLIENT_SECRET`, `EXTPROC_OAUTH2_CLIENT_CREDENTIALS_ENDPOINT`, `EXTPROC_OAUTH2_CLIENT_ASSERTION_TYPE` (default `id_token`). Behaviour (`exchanger.go`, `server.go`): on `RequestHeaders` it extracts the inbound `Authorization` Bearer (no bearer => **pass-through**), builds `resource = {scheme}://{authority}{path}`, mints its own client_assertion via a `client_credentials` grant (client_id/secret) and then POSTs RFC 8693 `grant_type=token-exchange` + `subject_token` + `resource` + `client_assertion` to the token endpoint; on success it **replaces upstream `Authorization: Bearer <exchanged token>`**; on `error_uri` it returns an HTTP-200 JSON-RPC `URLElicitationRequiredError` (code `-32042`); on infra failure it returns 500/503.
- **AIB broker chart** (`agentic-identity-broker/charts/agentic-identity-broker/`): deploys ONLY the main broker (single container, ports enduser `8000` + admin `14000`) + optional Postgres. **It does NOT deploy the ExtProc service** (no extproc template) and does not deploy the access-check gRPC service either (P2 used a separately-built deployment under `./tmp/security/build/`). The broker token endpoint already supports RFC 8693 token exchange, configured via `broker.tokenExchange` in `values.yaml` (`expectedAudience`, `claimExtraction.principalExpression`/`agentIdExpression`, `authorization.cel`). So P7 must (a) deploy the ExtProc service and (b) provision its OAuth client + a third-party service/grant in the broker.
- **CLI install** (`kaos-cli/kaos_cli/install.py`): under `--auth-enabled` it installs the broker (`_install_aib`, local chart), deploys the sync service (`_deploy_sync_service`), and wires the operator via `_build_auth_operator_args(ext_authz_url, issuer, credential_secret_prefix)` -> `--set security.agentAuth.*`. Port constants: `AUTH_EXT_AUTHZ_PORT=9191`, `AUTH_ENDUSER_PORT=8000`, `AUTH_ADMIN_PORT=14000`. Defaults: `_default_ext_authz_url` (`aib-access-check-grpc.<ns>...:9191`), `_default_auth_issuer`, `_default_auth_admin_url`. **No ExtProc deploy, no `extProcUrl` wiring, no third-party service registration.**

**Target (ADR-KAOS-009 + ADR-KAOS-004)**: `agentAuth.extProcUrl` (e.g. `aib-extproc.<ns>.svc.cluster.local:50051`) becomes the gateway `ext_proc` backend. The operator emits an `EnvoyExtensionPolicy` per protected route (gated, default-off) so token exchange runs **after** `ext_authz` in the same pass. The exchanged-for resource naming must match what AIB has policies/services for; for the MVP this is validated with a dedicated dummy third-party route/service rather than blanket-applied in a way that breaks internal traffic.

## Design plan (how it fits)

### Operator: `EnvoyExtensionPolicy` generation (new, parallel to SecurityPolicy)

- **New file `operator/pkg/security/envoyextensionpolicy.go`** mirroring `securitypolicy.go`:
  - `EnvoyExtensionPolicyGVK` = `gateway.envoyproxy.io/v1alpha1` kind `EnvoyExtensionPolicy`.
  - `constructEnvoyExtensionPolicy(params PolicyParams, cfg Config) (*unstructured.Unstructured, error)` — sets `spec.targetRefs` (identical shape to SecurityPolicy: group `gateway.networking.k8s.io`, kind `HTTPRoute`, name `params.RouteName`) and `spec.extProc` with the ext_proc backendRef built from `cfg.ExtProcBackendRef()`. **Encode the `spec.extProc` body to match the verified v1.4.6 schema** (request-headers processing mode, since the AIB ExtProc only mutates on `RequestHeaders`).
  - `ReconcileEnvoyExtensionPolicy(ctx, c, scheme, owner, params, cfg, log)` — same create-or-update + `SetControllerReference` flow as `ReconcileSecurityPolicy`; **no-op unless `cfg.ExtProcEnabled()`** (independent of `IsOperational()` so it can be enabled separately, but in practice gated together).
- **`config.go`**: add `ExtProcURL` field + env `SECURITY_AGENT_AUTH_EXT_PROC_URL`, read in `GetConfig()`; add `ExtProcEnabled()` (true when `ExtProcURL` is set) and `ExtProcBackendRef()` (reuse the same host:port -> name/namespace/port parser as `ExtAuthzBackendRef`; factor the shared parse into a private helper to avoid duplication).
- **Controllers**: in all three controllers, alongside the existing `ReconcileSecurityPolicy` call (still under `IsOperational()`), add a `ReconcileEnvoyExtensionPolicy(...)` call guarded by `secCfg.ExtProcEnabled()` using the same `PolicyParams`. Keep it a separate guard so token exchange is opt-in.
- **RBAC**: add `envoyextensionpolicies` (verbs create/delete/get/list/patch/update/watch) under the `gateway.envoyproxy.io` apiGroup in both `operator/config/rbac/role.yaml` and `operator/chart/templates/kaos-operator-rbac.yaml`. Add the kubebuilder RBAC marker next to the existing `securitypolicies` marker so `make manifests` keeps `role.yaml` in sync.

### Operator chart

- `values.yaml`: add `security.agentAuth.extProcUrl` (default empty).
- `operator-configmap.yaml`: render `SECURITY_AGENT_AUTH_EXT_PROC_URL` from `security.agentAuth.extProcUrl` (mirror the existing `extAuthzUrl` env), omitted when empty.

### AIB ExtProc deployment + broker provisioning (vendored chart + CLI)

- **Extend the vendored broker chart** (`agentic-identity-broker/charts/agentic-identity-broker/`) with an optional **ExtProc component** (default-off): a Deployment + Service (`aib-extproc`, gRPC `:50051`) running the `Dockerfile.extproc` image, configured via `EXTPROC_*` env from chart values (`extProc.enabled`, `extProc.image`, `extProc.oauth2.tokenEndpoint`/`issuer`/`clientId`/`clientSecret`/`clientAssertionType`). Wire the token endpoint default to the in-cluster broker enduser URL. This is the home for the service (the same place access-check should eventually live); keep it minimal and toggled.
- **CLI** (`install.py`): add `_default_ext_proc_url(auth_namespace)` = `aib-extproc.<ns>.svc.cluster.local:50051` (add `AUTH_EXT_PROC_PORT=50051`); under `--auth-enabled` enable the chart ExtProc component (`--set extProc.enabled=true` + its oauth2 values) when token exchange is requested; extend `_build_auth_operator_args(...)` to also append `--set security.agentAuth.extProcUrl=<url>` when set; add a programmatic broker-provisioning step (`_provision_token_exchange`) that registers the ExtProc OAuth client and a **dummy third-party service + delegated grant** via the broker admin API (so the exchange has something to exchange for in validation). Keep the third-party registration behind a dev/validation flag.
- New flags in `system/__init__.py`: `--token-exchange/--no-token-exchange` (default on under `--auth-enabled`), `--ext-proc-url`, threaded into `install_command`.

### Dummy third-party validation surface (scratch, ./tmp)

- Reuse the P2/P4 scratch pattern under `./tmp/security/extproc/`: a tiny "third-party" HTTP backend that echoes the received `Authorization` header, an HTTPRoute to it, and the `EnvoyExtensionPolicy` attached. Register a matching third-party service + delegated grant in the broker. This proves the exchange end-to-end without a real external IdP.
