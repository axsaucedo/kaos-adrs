## Problem statement

P2 delivered a **working, not production** AIB access-check: a decision domain service (`internal/domain/accesscheck`), an HTTP surface (`POST /api/access/check`), and an Envoy `ext_authz` gRPC `Check` surface, all keyed on the actor token and failing closed on missing/invalid actor. P11 hardens that slice to the production bar defined by ADR-AIB-002's functional/gateway/SDK requirements, so it is fit to upstream (the actual upstreaming is P14). The concrete gaps to close, all grounded in the current code: (1) there is **no subject (user) token handling at all** — the request, domain service, and response are actor-only, so FR-002/003/004/010 (validate the subject token when present, derive the subject principal, keep the decision strictly `actor -> resource`, never treat a `UserGrant` as resource access) are unmet; (2) the `/api/access/check` endpoint is registered **outside** the principal middleware and is therefore **unauthenticated** — any pod can request decisions, violating FR-005 (the API must validate the caller is authorized to request decisions); (3) the decision response is a thin `{allowed, reason}` with no `decision_id`, `principal`, `matched_grant`, or human-readable `message`, missing FR-007/014 and the ADR response contract; (4) there is **no structured audit logging** for allow/deny and no explicit token/secret **redaction** discipline, missing FR-012/013; (5) there is **no latency budget/timeout** on a decision and the fail-closed set is incomplete (e.g. ambiguous-resource mapping is not handled), missing FR-006/015; (6) the endpoint is **not documented in the AIB OpenAPI** (`api/enduser/openapi.yaml`), violating the AIB Constitution's API-first principle, and the **synthetic-service/PermissionSet bootstrap encoding is not formally labeled temporary nor is the target first-class resource-grant model evaluated** (FR-009/011); (7) there are **no metrics** and the productionised surfaces have not been re-validated end to end in KIND. None of this changes the API's external *shape* beyond the additive response/contract fields; it makes the existing contract correct, safe, observable, and documented.

## Current-state grounding (researched, on `feat/aib-access-check`)

- **Domain** (`internal/domain/accesscheck/service.go`): `Service.Decide(ctx, actorID id.AgentID, resource, action string) (Decision, error)`; `Decision{Allowed bool, Reason string}`; reasons `granted`/`grant_missing`/`resource_unknown`/`agent_unknown`. Resolves agent -> `agent.PermissionSets` -> `permSets.GetByIDs` -> `covers(permSets, serviceID, action)`; maps resource via `resourceClientID("kaos://{kind}/{ns}/{name}" | "kaos://{kind}/{id}") -> "kaos-{...}"` then `ServiceResolver.FindByClientID`. **No subject parameter; no decision_id; no matched_grant.** Fails closed by returning a deny on not-found and propagating repo errors up (callers translate to deny/500).
- **HTTP** (`internal/adapters/http/handlers/accesscheck/handler.go`): `checkRequest{actor_token, resource, action}` (**no subject_token**); `checkResponse{allowed, reason, actor, resource, action}` (**no principal/decision_id/matched_grant/message**); always 200 for decisions, 400 for malformed body/missing resource, 500 on decider error; actor token from `actor_token` body or `x-agent-authorization` header (Bearer-stripped); invalid actor -> `reason: invalid_actor_token`, allowed=false, 200.
- **Caller authz gap** (`internal/adapters/http/routing/enduser.go:87-88`): `r.Post("/access/check", h.AccessCheck.CheckAccess)` is registered on the **public** router, NOT under the `authRouter.Use(RequirePrincipalMiddleware(...))` block (lines 94/131). So the endpoint currently authenticates the *actor token in the body* but not the *caller*. Pre-auth uses `RequirePrincipalMiddleware(cfg.Authentication, cfg.JWTAuthenticator, ...)` and the `X-Remote-User` plain-header pre-auth (ADR-AIB-000).
- **gRPC** (`internal/adapters/grpc/accesscheck/server.go`): `Check(ctx, *authv3.CheckRequest)`; resource/action from `context_extensions` (`kaos.resource`/`kaos.action`) or `x-kaos-resource`/`x-kaos-action` headers; actor from `x-agent-authorization`; allow -> `OkHttpResponse{}` (**no decision metadata headers added -> GW-006 gap**); deny -> `DeniedHttpResponse` 403 with `x-kaos-access-reason` header. No subject handling; no decision_id.
- **Actor token** (`internal/adapters/accesscheck/actor_token.go`): `ActorTokenValidator` over the broker's published JWKS (`KeySetProvider.GetKeySet`), signature + temporal validation, optional `expectedIssuers` (wired to `localIssuerURI` in `builder.go:935`), 60s skew, returns `id.AgentID`, fails closed (`ErrInvalidActorToken`). **There is no subject/user-token validator wired into access-check** — the broker's user-token verification building blocks (jwtauth / RequirePrincipalMiddleware's `JWTAuthenticator`) exist separately and must be reused for the subject token.
- **Wiring** (`internal/app/builder.go:926-967`): access-check is gated (nil when disabled); constructs `ActorTokenValidator(publishedJWKS, [localIssuerURI])`, `ServiceResolver(storage.Services())`, `domaccesscheck.NewService(...)`, `NewHandler(...)`; exposes `AccessCheckService`/`AccessCheckActorValidator` on `app`. `internal/app/handlers.go:55` holds the handler.
- **Authz semantics to honour** (ADR-KAOS-005 Decision): decision is keyed on the **ACTOR** (`x-agent-authorization`); the user **SUBJECT** (`Authorization`) is used only for user-delegated third-party grants, never for resource-to-resource attribution; autonomous runs are actor-only; data-first (requested edge vs approved resource grant vs delegated user grant). Bootstrap mode (ADR-AIB-002 FR-009) encodes platform grants as a synthetic service + PermissionSet, kept distinct from `UserGrant`, explicitly temporary.
- **OpenAPI**: `api/enduser/openapi.yaml` and `api/admin/openapi.yaml` exist; the access-check endpoint is on the **enduser** surface and is currently undocumented.

## Design plan (how it fits)

### Subject (user) token handling — actor/subject separation (FR-002/003/004/010)

- Extend the request contract additively with an optional `subject_token` (HTTP body field; gateway path reads the user token from the standard `Authorization` header alongside the actor's `x-agent-authorization`). Reuse the broker's **existing user-token validation** (the `JWTAuthenticator` already used by `RequirePrincipalMiddleware`) — do **not** validate the subject with the actor validator (different issuer/audience). When a subject token is present, validate it (signature/issuer/audience/exp/nbf) and derive the **subject principal**; when absent (autonomous run), proceed actor-only. Thread the subject principal into `Service.Decide` as an optional argument and into the response/audit. The decision **stays strictly `actor -> resource`** from permission-set coverage; the subject principal does **not** by itself grant resource access (FR-010) — it is recorded for audit and reserved for user-delegated third-party grant evaluation (which token exchange, not this API, performs). A malformed/invalid subject token when one is supplied fails closed (deny), since presenting an invalid identity is not a valid request.

### Caller authorization (FR-005)

- Move the `/access/check` route **under** an authorization boundary so an unauthenticated caller cannot request decisions. Two callers exist: the **gateway** (gRPC `ext_authz`, trusted network boundary — already the gateway's own identity) and **SDK/off-gateway** callers (HTTP). For the HTTP surface, require the caller to be authenticated via the broker's pre-auth principal (`RequirePrincipalMiddleware` / `X-Remote-User`) and authorize it against a configured allowed-caller set (e.g. the sync/gateway service principal), failing closed (`401`/`403` at the request layer, distinct from a 200 deny *decision*). Keep this gated by config so a dev/no-auth install still works, but default to requiring a caller principal when access-check is enabled in a secured posture. Document that the gRPC surface's caller is the gateway at a trusted boundary.

### Structured decision contract: decision_id + matched_grant + principal + message (FR-007/014)

- Enrich `Decision` with `DecisionID` (stable, generated per evaluation — UUID/ULID, also accepting a caller-supplied idempotency/correlation id when provided), `MatchedGrant` (type + id when allowed, e.g. the covering PermissionSet/service), and `Message` (human-readable). Align the reason taxonomy to the ADR (`granted`, `grant_missing`, `resource_unknown`, `agent_unknown`, `invalid_actor_token`, plus a subject-related reason when a supplied subject token is invalid). Surface the full response on both surfaces: HTTP body `{allowed, principal, actor, resource, action, decision_id, matched_grant, reason, message}`; gRPC propagates `decision_id`/`reason` (and, on allow, safe metadata — see latency/GW-006 below).

### Audit logging + redaction (FR-012/013)

- Emit one structured audit log line per decision for **both** allow and deny (fields: `decision_id`, `actor`, `subject`/principal, `resource`, `action`, `reason`, `matched_grant`, latency) at the domain/handler boundary. Introduce/centralise a redaction helper guaranteeing raw `actor_token`/`subject_token`/secrets **never** appear in logs, traces, or error messages anywhere in the access-check path (handler decode errors, validator errors, gRPC). Errors returned to callers are generic ("internal server error"); details go to logs without token material.

### Latency budget, fail-closed completeness, GW-006 metadata (FR-006/015, GW-006)

- Apply a configurable per-decision deadline (gateway-friendly budget, e.g. default ~250ms via `context.WithTimeout`) so a slow repo/JWKS call fails closed rather than stalling the gateway. Complete the fail-closed set: missing/invalid actor, malformed actor/subject tokens, missing resource, **ambiguous resource** (a resource URI that resolves to more than one service, or a malformed multi-segment URI), repository errors, and policy/evaluation errors all deny (200 `allowed:false` for the HTTP decision surface; 403 for the gRPC gateway surface). On the gRPC **allow** path, add safe, non-sensitive decision metadata as **set-not-append** request headers to the backend (`x-kaos-actor`, `x-kaos-principal` when present, `x-kaos-decision-id`) per GW-006 — never the raw tokens.

### OpenAPI + bootstrap labeling + target-model evaluation (FR-009/011, Constitution IV)

- Document `POST /api/access/check` in `api/enduser/openapi.yaml`: request (`actor_token`, optional `subject_token`, `resource`, `action`), the always-200-for-decisions behaviour, and the full response schema including `decision_id`/`matched_grant`/`principal`/`message`, plus the 400/401/403 request-layer errors. Explicitly **label the synthetic-service/PermissionSet bootstrap encoding as temporary** in code comments and a short design note, keeping it distinct from `UserGrant` (FR-009/010). Produce a written **evaluation** (a spec/design note under the AIB `specs/` or `docs/` tree, following AIB conventions) recommending the path to a first-class platform/resource-grant model (`PlatformResource`/`ResourceGrant`/`RequestedAccessEdge`/`ResourceAccessDecision`) behind the **unchanged** API contract (FR-011 target mode): what the model is, how `Decide` internals swap from bootstrap lookup to native grants without an API change, and the migration. Implement first-class scaffolding only if low-risk within this phase; otherwise the evaluation is the deliverable and implementation defers (it is "evaluate", not "implement", per the split).

### Metrics + observability

- Add access-check metrics following AIB telemetry conventions: a decision counter labelled by `allowed`/`reason`, an error counter, and a decision-latency histogram. Wire through the existing telemetry/meter provider used elsewhere in the broker.
