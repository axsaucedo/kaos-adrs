## Problem statement

The prior phases stood up gateway enforcement (P2 ext_authz, P6 user-auth/`jwt_authn`, P7 ext_proc token exchange), bypass prevention (P4 ingress NetworkPolicy + TLS modes), and the sync/SDK machinery. What is missing is the **operational correctness story**: when enforcement *denies* or *needs a human action*, the system must surface a **structured, machine-readable outcome** end to end, and the secured posture must be production-hardened. P12 closes this, realising ADR-KAOS-006 (re-authentication execution model), the remainder of ADR-KAOS-005 (requested-vs-approved + backend-neutral OPA), and the hardening of ADR-KAOS-007 (NetworkPolicy/TLS). Concretely: (1) AIB must emit the ADR-006 outcome taxonomy — `platform_grant_missing` and `user_grant_required` on the `ext_authz` denied path, and `third_party_reauth_required` + `reauth_url` on the `ext_proc` token-exchange immediate path — instead of the generic `grant_missing`; (2) the **runtime must consume** these outcomes: a synchronous call surfaces the reason (and re-auth URL when present) to the caller without blocking, and an **autonomous run records a structured `user_action_required` event and stops/skips** the protected action — never blocking indefinitely, never auto-granting; (3) the **requested-vs-approved** semantics must be finalised end to end — a CRD reference is a *requested* edge only and must never auto-create an *approved* grant, so the gateway denies until an admin approves; (4) the authorization integration must be proven **backend-neutral** by validating an **OPA (opa-envoy) drop-in** over the same `ext_authz` contract with no Envoy/KAOS config change; (5) the transport posture must be hardened — **egress** NetworkPolicy (deferred from P4) plus TLS minimum-version/cert-path validation. The runtime today has **no re-auth handling at all** and AIB emits only `grant_missing`, so this is net-new surfacing built on existing mechanisms, default-gated so non-secured installs are unchanged.

## Current-state grounding (researched)

- **ADR-006 outcome routing** (ADR-KAOS-006 Decision table): `platform_grant_missing` -> `ext_authz` denied, **no** user-action URL; `user_grant_required` -> `ext_authz` denied, no dedicated renewal URL field in current AIB; `third_party_reauth_required` + `reauth_url` -> `ext_proc` immediate response from token exchange (`error_uri`/re-auth URL); autonomous-run-needs-action -> record a structured `user_action_required` event and stop/skip. No blocking waits, no durable pause/resume, no auto-grant. A2A `input-required` durable resume and tool/argument approvals are deferred.
- **AIB access-check reasons** (`internal/domain/accesscheck/service.go`, after P11): `granted`/`grant_missing`/`resource_unknown`/`agent_unknown`/`invalid_actor_token`; decision is actor->resource platform coverage via permission sets. The gRPC `ext_authz` `Check` (`internal/adapters/grpc/accesscheck/server.go`) denies with `403` + `x-kaos-access-reason` header; allow returns `OkHttpResponse`. `grant_missing` must be split into `platform_grant_missing` (no actor->resource platform grant) and, where a resource requires a user-delegated grant, `user_grant_required` (reuse the broker's `consent.VerifyAgentAccess` building block). `consent`/user-delegated grants are otherwise evaluated at **token exchange** (ext_proc), not access-check.
- **AIB ext_proc** (token-exchange service, used by P7): already builds an HTTP resource URI and performs RFC 8693 exchange, and per P7 maps a third-party `error_uri` to an immediate response (JSON-RPC `-32042`). P12 formalises this to carry `reason: third_party_reauth_required` + `reauth_url` in the machine-readable shape the runtime parses. Verify the exact current shape before changing.
- **Runtime (pais)**: `pais/server.py` instruments `httpx` via the AIB SDK (`aib.instrument_httpx()`, line 148) so outbound calls to MCP/ModelAPI/other agents flow through the gateway and receive `ext_authz`/`ext_proc` responses. There is **no** re-auth/outcome handling today (grep for `reauth`/`grant_missing`/`user_action` in `pais/` returns nothing). The P9 SDK `Client` exposes typed re-auth/auth outcomes and a single-401 retry — P12 layers `403`/ext_authz-reason and ext_proc-reauth interpretation on top. `pais/a2a.py`: `TaskState` enum includes `INPUT_REQUIRED = "input-required"` (line 52); `Task` has `events: List[TaskEvent]` + `add_event` + `metadata` dict (lines 154-186); `_execute_autonomous` (line 531) runs the autonomous loop with a per-iteration `try/except` (`iter_err`, line 664) — the insertion point for recording `user_action_required` and stopping/skipping the protected action.
- **NetworkPolicy** (`operator/pkg/security/networkpolicy.go`, from P4): typed `networking/v1`, **ingress-only**, allow-from gateway + operator namespaces, no-op unless `cfg.NetworkPolicyEnabled()` (`SECURITY_NETWORK_POLICY_DISABLED`). Egress was **explicitly deferred to P12** (P4 plan, ingress-only decision) because ModelAPI proxies egress to external LLM providers and workloads need DNS/registry/gateway/AIB egress. Reconciled in the three controllers next to `ReconcileSecurityPolicy` inside `if secCfg.IsOperational()`. Workload pod labels: Agent `{app:agent,agent:name}`, MCPServer `{app:mcpserver,mcpserver:name}`, ModelAPI `{app:modelapi,modelapi:name}`.
- **TLS** (`operator/chart/templates/gateway.yaml` + `gateway-tls.yaml`, from P4): `security.tls` block with `selfSigned`/`certManager`/`provided` modes -> HTTPS/443 listener + `kubernetes.io/tls` Secret. No minimum-TLS-version pinning yet. mTLS/SPIFFE/mesh/native intra-service TLS are **out of scope (not deferred)** by ADR-KAOS-007.
- **Requested-vs-approved** (ADR-KAOS-005 Decision, points 12/decision-summary): data-first — *requested edge* (CRD reference) vs *approved resource grant* (AIB, `status: approved`) vs *delegated user grant*. "An Agent that references an MCPServer in its CRD would not automatically be allowed to call it." The sync (P5) projects requested edges **kept distinct** from approved grants; the gateway enforces only approved. P12 finalises/guards this (no auto-approve anywhere) and proves it end to end.
- **OPA drop-in** (ADR-KAOS-005, lines 61-67): the integration is backend-neutral over `envoy.service.auth.v3.Authorization/Check` with neutral `kaos.resource`/`kaos.action` `context_extensions`; "OPA is swappable as a drop-in backend via opa-envoy with no Envoy or KAOS configuration change, provided it has equivalent policy and data." P2's operator generates the `SecurityPolicy` `extAuth.grpc` pointing at the configured backend — pointing it at opa-envoy instead of AIB requires only config (the backend ref), no code change.
- **Test/build**: KAOS — operator `make test-unit`/`go test ./pkg/security/...`/`make manifests generate`; runtime `cd pydantic-ai-server && uv run pytest tests/ -v` + `make lint` (black + `ty`); CLI `pytest` + `make lint`; chart `helm template`/`helm lint`. AIB — `just test` (`-race`), KIND `kind-image`.

## Design plan (how it fits)

### A. AIB outcome taxonomy (ext_authz + ext_proc) — TODO 1

Split the access-check `grant_missing` into `platform_grant_missing` (the actor has no approved platform grant for the resource) and `user_grant_required` (the resource requires a user-delegated grant that is missing/expired — detected via the existing `consent.VerifyAgentAccess` building block; only where applicable, otherwise the platform reason stands). Surface the reason on both surfaces: the HTTP body (P11 contract) and the gRPC `ext_authz` denied response (`x-kaos-access-reason` header + body) so the gateway forwards a machine-readable code. Formalise the ext_proc token-exchange immediate response to carry `reason: third_party_reauth_required` + `reauth_url` (from the AIB OAuth2 session `error_uri`) in the same JSON-RPC error shape the runtime parses. `platform_grant_missing` carries **no** URL (ADR-006). This is the only AIB-repo task; everything else is KAOS.

### B. Runtime synchronous outcome consumption — TODO 2

On the runtime's outbound path (httpx, already SDK-instrumented), interpret the gateway-surfaced outcomes into typed results using the P9 SDK `Client` outcome types: an `ext_authz` `403` with `x-kaos-access-reason` -> a typed authorization outcome (`platform_grant_missing`/`user_grant_required`); an `ext_proc` immediate `third_party_reauth_required` body -> a typed re-auth outcome carrying `reauth_url`. Map these to a structured error surfaced in the synchronous chat/A2A response (e.g. the assistant message states the denied resource + reason, and "reconnect <service> at <reauth_url>" when a URL is present) **without blocking** and **without auto-granting**. A `user_grant_required`/`platform_grant_missing` with no URL surfaces guidance to request/approve the grant. Keep this additive: when security is off / no such response is seen, behaviour is unchanged.

### C. Autonomous `user_action_required` event — TODO 3

In `_execute_autonomous`, when a protected action raises one of the typed outcomes from (B), **record a structured `user_action_required` event** on the task (`TaskEvent` with `reason`, `resource`, `reauth_url?`, `action`) and **stop/skip** that protected action for the iteration — do not block, do not retry indefinitely, do not auto-grant. The run continues (or ends per the existing loop policy) rather than hanging. Optionally reflect it in `Task.metadata`/state for observability, but do **not** use A2A `input-required` durable pause/resume (deferred by ADR-006). Surface the event in the task's `events` so an operator/API can see what human action is needed.

### D. Requested-vs-approved finalisation — TODO 4

Finalise and guard the no-auto-grant invariant end to end: (1) the **sync** projects CRD references as *requested* edges and must never create an `approved` resource grant from a CRD reference alone (verify P5's projection keeps requested distinct from approved; add a guard/test if any path could approve); (2) the **operator** never writes approval state; (3) prove with a test that an Agent referencing an MCPServer is **denied** at the gateway until an admin approves the grant in AIB, and **allowed** after. This is the runtime realisation of "no-permission-by-default". Where the bootstrap encoding represents grants, ensure requested vs approved remain separable (ties to P11's bootstrap labeling).

### E. Backend-neutral OPA drop-in validation — TODO 5

Demonstrate (not default) that OPA via **opa-envoy** serves the same `ext_authz` contract: provide a sample Rego policy + data bundle that consumes equivalent grant data and decides `allow`/`deny` from `kaos.resource`/`kaos.action` + the actor identity, deployed as an opa-envoy gRPC `ext_authz` backend. Point the operator-generated `SecurityPolicy` `extAuth.grpc` at the opa-envoy service via **configuration only** (the existing backend-ref config), with no Envoy/KAOS code change, and show equivalent allow/deny in KIND. Deliverable = a sample policy/manifest under `docs`/samples + a documented validation + a test asserting the SecurityPolicy backend ref is configurable. AIB stays the default backend.

### F. Egress NetworkPolicy + TLS hardening — TODO 6

Add the **egress** dimension to `networkpolicy.go` (deferred from P4): extend `constructNetworkPolicy` to `PolicyTypes: [Ingress, Egress]` when egress hardening is enabled, with egress rules allowing **DNS** (kube-dns/UDP+TCP 53), the **Gateway** data-plane, **AIB**, and (for ModelAPI workloads) **external LLM provider** egress, default-deny otherwise. Make egress **separately gated** (`SECURITY_NETWORK_POLICY_EGRESS_ENABLED`, default conservative) so it can't silently break ModelAPI proxy traffic; the ModelAPI policy must permit outbound to external providers (likely allow all egress except restricting to the gateway for in-cluster, or an explicit provider allowance — choose the safe default and document). TLS hardening: pin a minimum TLS version on the HTTPS listener and validate the cert path renders for all three modes. Gate everything; non-secured/dev installs unchanged.

### G. Validation, docs, REPORT — TODO 7

End-to-end on a **Calico** KIND cluster (egress + OPA need real NetworkPolicy/ext_authz enforcement): prove the full outcome story (deny -> structured reason in the sync response; third-party -> reauth_url; autonomous -> `user_action_required` event + stop; requested-only -> denied until approved -> allowed; OPA drop-in equivalent; egress default-deny with allowed paths working; TLS terminated with min version). Docs across the touched surfaces; push PRs; CI.

## Out of scope (later phases)

- A2A `input-required` durable pause/resume, runtime tool/argument approval workflows, and runtime-created admin approval queues — deferred by ADR-KAOS-006 (KAOS lacks durable tasks/resume + tool-permission model).
- mTLS, SPIFFE, service mesh, native intra-service (pod-to-pod) TLS, cryptographic workload binding — out of scope (not deferred) by ADR-KAOS-007.
- Making OPA the default authorization backend or building a generalized PDP envelope/policy-plugin layer — ADR-KAOS-005 keeps AIB grants the default; OPA is a validated optional drop-in only.
- Implementing a first-class platform/resource-grant data model in AIB (P11 evaluates it; upstreamed in P14) and upstreaming AIB via fork (P14).
- Documentation consolidation across all components beyond the per-surface docs this phase requires — P13.
